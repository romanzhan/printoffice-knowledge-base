Интеграции

Полный реестр внешних систем, с которыми интегрируется CRM/сайт типографии PrintOffice: мессенджеры, email, ЭДО, бухгалтерия, облачное хранилище, курьерские службы, платёжные сервисы, аналитика, система пропусков — с назначением, известными деталями API и открытыми вопросами.

1. Мессенджеры (Макс / Telegram)#

1.1. Проблема и цель#

Переписка с клиентами идёт в разных чатах (Макс, Telegram, WhatsApp). Информация дублируется, единого канала нет. Цель — интегрировать мессенджер (Макс или Telegram) с CRM, чтобы вся коммуникация была в одном месте.

Типовой текущий путь клиента: клиент приходит с сайта (из поиска Яндекс/Google, Яндекс.Карт, 2ГИС, Google Карт, Яндекс.Директа), затем переходит по ссылке в Telegram-чат с менеджером, где ведётся диалог, расчёты, уточнение сроков, ТЗ и стоимости. Желаемое изменение: перенести расчёт стандартных заказов на сайт/бот, освободив менеджера от рутины и подключая его только когда ресурсов сайта/бота недостаточно.

1.2. Что нужно видеть в CRM из мессенджеров#

  • Историю сообщений (для аналитики).
  • Возможность отправлять сообщения прямо из CRM (требуется бот).
  • Автоматическое создание заказа по сообщению (например, клиент написал «Хочу листовки» → CRM создаёт сделку).

⚠️ ОТКРЫТЫЙ ВОПРОС: Какой мессенджер является основным каналом для общения с клиентами — Telegram, WhatsApp или Макс (специализированное приложение)? Это определяет приоритет и техническую реализацию интеграции.

⚠️ ОТКРЫТЫЙ ВОПРОС: Наличие и возможности публичного API у мессенджера Макс не подтверждены. Если API нет — интеграция реализуется через вебхуки.

2. Telegram-бот#

Telegram — один из ключевых каналов коммуникации. Бот @print_bot обрабатывает первый контакт, выполняет расчёт стоимости, показывает заказы клиента и связывает с менеджером.

2.1. Общая схема первого контакта#

  1. Клиент находит сайт, видит кнопку «Telegram» и кликает.
  2. Открывается Telegram, запускается бот @print_bot.
  3. Бот отправляет приветствие (текст + кнопки).
  4. Далее три ветки:
    • Клиент пишет текстом («здравствуйте») → менеджер отвечает вручную.
    • Клиент нажимает «Рассчитать» → калькулятор в Telegram (WebView).
    • Клиент нажимает «Мои заказы» → авторизация через сайт.

2.2. Сценарии бота#

Инициализация. Бот отправляет приветственное сообщение:

Здравствуйте! Я бот типографии PrintOffice. Чем могу помочь? ✅ Рассчитать стоимость печати 📦 Мои заказы 📞 Связаться с менеджером ❓ Часто задаваемые вопросы

И показывает клавиатуру с кнопками: Рассчитать стоимость, Мои заказы, Связаться, FAQ, Оставить заявку, Статус заказа.

Сценарий «Рассчитать стоимость»:

  1. Бот открывает WebView (встроенный браузер) по URL https://printoffice.ru/calculator?tg=1&user_id={user_id}.
  2. Клиент видит калькулятор (как на сайте): выбирает тип продукции, формат, бумагу, цветность, вводит тираж, нажимает «Рассчитать».
  3. Калькулятор показывает цену.
  4. Клиент нажимает «Отправить заявку» → заявка попадает в CRM (статус «Новая (из Telegram)», привязка к chat_id клиента, уведомление менеджеру).
  5. Бот отправляет подтверждение: «Ваша заявка №123 принята. Менеджер свяжется с вами в этом чате.»

Сценарий «Клиент пишет текстом» (NLP / ключевые слова):

  • «здравствуйте», «цену», «сколько стоит» → распознавание намерения.
  • «печать», «листовки», «визитки» → распознавание продукта.
  • «1000», «500», «тираж» → распознавание тиража.

Логика обработки:

  • IF распознан запрос на расчёт → перейти к калькулятору.
  • ELSE IF распознан запрос статуса заказа → запросить номер заказа.
  • ELSE → перевести на менеджера.

Примеры ответных реплик бота:

Я вас понял. Давайте рассчитаем стоимость. Какой формат вам нужен? (A4, A5, A6) Соединяю с менеджером...

Сценарий «Мои заказы»:

  • Бот проверяет авторизацию (связан ли chat_id с аккаунтом).
  • IF авторизован → показать список заказов.
  • ELSE → запросить привязку аккаунта по ссылке https://printoffice.ru/link_tg?chat_id={chat_id}.

Пример списка заказов:

№12345 | Листовки А4 | 1000 шт | 15.03.2026 | ✅ Готов
№12346 | Визитки     | 500 шт  | 20.03.2026 | 🚚 В доставке
№12347 | Брошюры     | 50 шт   | 25.03.2026 | ⏳ В работе

2.3. Архитектура бота#

  • Telegram Bot API (BotFather) → Webhook (получение сообщений) → Backend (обработка).
  • PostgreSQL — chat_id, история сообщений.
  • Redis — сессии, кэш.
  • NLP-модуль — распознавание намерений.
  • Калькулятор — расчёт стоимости.
  • WebView — встроенный браузер для калькулятора.
  • CRM — заказы.

2.4. Структура БД для Telegram#

-- Связь пользователя с Telegram
CREATE TABLE tg_connections (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_id INT NOT NULL,                    -- ID пользователя в CRM
    chat_id BIGINT NOT NULL,                 -- Telegram chat_id
    username VARCHAR(255),                   -- Telegram username
    first_name VARCHAR(255),
    last_name VARCHAR(255),
    phone_number VARCHAR(20),
    is_active BOOLEAN DEFAULT true,
    linked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    last_activity TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(id),
    UNIQUE KEY unique_chat (chat_id),
    UNIQUE KEY unique_user (user_id)
);

-- История сообщений
CREATE TABLE tg_messages (
    id INT PRIMARY KEY AUTO_INCREMENT,
    chat_id BIGINT NOT NULL,
    message_id INT NOT NULL,                 -- ID сообщения в Telegram
    direction ENUM('incoming', 'outgoing') NOT NULL,
    message_text TEXT,
    intent VARCHAR(50),                      -- распознанное намерение
    entities JSON,                           -- извлечённые сущности
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    INDEX idx_chat (chat_id, created_at)
);

-- Сессии диалогов
CREATE TABLE tg_sessions (
    id INT PRIMARY KEY AUTO_INCREMENT,
    chat_id BIGINT NOT NULL,
    current_state VARCHAR(50),               -- 'awaiting_format', 'awaiting_quantity'
    context JSON,                            -- {format: "A4", quantity: 1000}
    expires_at TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_chat (chat_id),
    INDEX idx_expires (expires_at)
);

2.5. Обработка сообщений (псевдокод/JS)#

class TelegramBotHandler {
    constructor() {
        this.botToken = process.env.TELEGRAM_BOT_TOKEN;
        this.webhookUrl = process.env.WEBHOOK_URL;
    }

    // Обработка входящего сообщения
    async handleMessage(chatId, messageText, messageId) {
        // 1. Сохраняем сообщение в историю
        await this.saveMessage(chatId, messageId, 'incoming', messageText);
        // 2. Распознаём намерение
        const intent = await this.recognizeIntent(messageText);
        // 3. Получаем текущую сессию
        let session = await this.getSession(chatId);
        // 4. Обрабатываем в зависимости от намерения
        let response;
        switch (intent.intent) {
            case 'greeting':          response = this.getGreetingResponse(); break;
            case 'price_calculation': response = await this.handlePriceCalculation(chatId, intent.entities, session); break;
            case 'order_status':      response = await this.handleOrderStatus(chatId, intent.entities); break;
            case 'my_orders':         response = await this.handleMyOrders(chatId); break;
            case 'contact_manager':   response = this.getContactManagerResponse(); break;
            case 'faq':               response = this.getFaqResponse(); break;
            default:                  response = this.getDefaultResponse();
        }
        // 5. Отправляем ответ
        await this.sendMessage(chatId, response.text, response.keyboard);
        // 6. Сохраняем ответ в историю
        await this.saveMessage(chatId, null, 'outgoing', response.text);
        // 7. Обновляем сессию
        await this.updateSession(chatId, intent.nextState, intent.context);
    }
}

const keywords = {

    product:  ['печать', 'листовки', 'визитки', 'брошюры', 'наклейки', 'буклеты'],
    quantity: ['тираж', 'штук', 'экземпляр', 'шт'],

Логика распознавания: приветствие → intent greeting; запрос цены → intent price_calculation (если формат не извлечён, nextState = awaiting_format); запрос заказа → если найден номер, intent order_status, иначе my_orders; прочее → unknown.

2.6. Расчёт стоимости в боте (handlePriceCalculation)#

  • Если есть все параметры (формат + тираж) → рассчитываем цену и показываем карточку с общей суммой и ценой за 1 шт, кнопками «✅ Оформить заказ» и «📞 Связаться с менеджером».
  • Если нет формата → предлагаем выбор: A4 (210×297 мм), A5 (148×210 мм), A6 (105×148 мм), «Свой размер».
  • Если нет тиража → предлагаем: 100, 500, 1000, 5000 шт, «Ввести свой».

2.7. Отправка сообщения (Telegram API)#

async sendMessage(chatId, text, keyboard = null) {
    const url = `https://api.telegram.org/bot${this.botToken}/sendMessage`;
    const payload = { chat_id: chatId, text: text, parse_mode: 'Markdown' };
    if (keyboard) payload.reply_markup = JSON.stringify(keyboard);
    const response = await fetch(url, {
        method: 'POST',
        headers: { 'Content-Type': 'application/json' },
        body: JSON.stringify(payload)
    });
    return response.json();
}

async function createOrderFromTelegram(chatId, orderData) {
    // 1. Находим или создаём пользователя
    let user = await findUserByChatId(chatId);

3. Email (IMAP / SMTP, парсер входящих)#

3.1. Назначение#

Приём заказов по email (от клиентов и от партнёра ГМ Принт), автоматический парсинг входящих писем с созданием сделки в CRM, отправка уведомлений и закрывающих документов.

3.2. Общая схема обработки email-заказов#

  1. Клиент отправляет файлы на print@printoffice.ru (тема, например, «Заказ на печать»).
  2. Система автоматически: распознаёт email, извлекает вложения, создаёт сделку в CRM, привязывает файлы к сделке.
  3. Менеджер получает уведомление о новой сделке.
  4. Менеджер определяет тип клиента: B2C (частное лицо) / B2B (организация).
  5. Менеджер делает расчёт (формат, бумага, цветность, тираж, доп. услуги).
  6. Берёт оплату: B2C — ссылка / QR-код; B2B — счёт на email.
  7. Передаёт заказ в производство.
  8. Сообщает о готовности (обычно на следующий день, при срочности — в день заказа).
  9. Выдача: самовывоз из офиса или доставка.
  10. Закрывает сделку (статус completed).

Детальный event map: обработка менеджером#

Логика ручной обработки сделки менеджером (граничные условия из event map):

  • Просмотр файлов (макетов):
    • IF макет в правильном формате → OK.
    • ELSE IF макет требует конвертации → конвертируем.
    • ELSE → запросить правильный формат у клиента.
  • Определение типа клиента:
    • IF клиент новый → уточнить, физическое или юридическое лицо:
      • B2C: запросить ФИО, телефон.
      • B2B: запросить название организации, ИНН, контактное лицо.
    • ELSE IF клиент существующий → берём из базы.
  • Расчёт: калькулятор в CRM, ввод параметров (формат, бумага, цветность, тираж) → система показывает цену.
  • Отправка расчёта клиенту:
    • IF B2C → ссылка на оплату в ответном email.
    • IF B2B → счёт на email.
  • Ожидание оплаты:
    • IF оплачено → запуск производства.
    • ELSE → напоминание через 24 часа.
  • Далее: производство выполняет заказ → менеджер сообщает о готовности → клиент забирает (самовывоз) или получает доставку → менеджер закрывает сделку (статус «Выполнен»).

3.3. Источники получения писем#

  • IMAP/POP3-подключение.
  • Webhook от почтового сервиса (SendGrid, Mailgun).
  • Яндекс.Почта API.

3.4. Извлекаемые данные#

  • Отправитель: email, имя (если есть).
  • Тема письма, тело письма.
  • Вложения (макеты PDF, CDR, AI, PNG, JPG) — сохраняются в S3 / облачное хранилище.

Поиск клиента: IF email найден в БД → привязываем к существующему; ELSE → создаём нового (тип «не определён»).

При создании сделки (лида) в CRM проставляются:

  • Статус: «Новый (из email)».
  • Источник: «email».
  • Приоритет: «срочный», если в теме есть «срочно», иначе «обычный».
  • Вложения: привязаны к сделке.
  • Параметры: (если удалось распознать) — попытка автораспознавания параметров заказа (формат, тираж, бумага) из тела письма.

CREATE TABLE incoming_emails (
    id INT PRIMARY KEY AUTO_INCREMENT,
    message_id VARCHAR(255) UNIQUE,          -- ID письма в почтовом ящике
    from_email VARCHAR(255) NOT NULL,
    from_name VARCHAR(255),
    subject VARCHAR(500),
    body TEXT,
    received_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    status ENUM('new', 'processing', 'processed', 'failed') DEFAULT 'new',
    deal_id INT,
    error_message TEXT,
    INDEX idx_status (status),
    INDEX idx_from (from_email),
    INDEX idx_received (received_at)
);

-- Таблица вложений
CREATE TABLE email_attachments (
    id INT PRIMARY KEY AUTO_INCREMENT,
    email_id INT NOT NULL,
    filename VARCHAR(255) NOT NULL,
    file_size INT,
    file_type VARCHAR(100),
    file_url VARCHAR(500),                   -- ссылка в S3
    is_maket BOOLEAN DEFAULT false,          -- является ли макетом
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (email_id) REFERENCES incoming_emails(id),
    INDEX idx_email (email_id)
);

-- Связь email с клиентом
ALTER TABLE clients ADD COLUMN email VARCHAR(255);
ALTER TABLE clients ADD COLUMN email_verified BOOLEAN DEFAULT false;
CREATE INDEX idx_client_email ON clients(email);

3.7. Интерфейс менеджера для обработки email-сделки#

Карточка сделки (пример — «Сделка №12345 (из email)», статус «Новая») содержит блоки:

  • Информация о письме: От (напр. Иван Иванов <ivan@example.com>), Тема (Заказ на печать листовок), Получено (15.03.2026 14:30), Приоритет (🔴 СРОЧНО).
  • Вложения: список файлов с действиями, например:
    • 📄 maket.pdf (2.5 MB) — СКАЧАТЬ, ПРЕДПРОСМОТР, ОТКРЫТЬ В РЕДАКТОРЕ.
    • 📄 logo.png (0.5 MB) — СКАЧАТЬ.
  • Клиент: Имя (Иван Иванов), Email (ivan@example.com), Телефон, Тип клиента (Не определён / B2C / B2B), кнопка СОХРАНИТЬ КЛИЕНТА.
  • Параметры заказа: Продукция (Листовки), Формат (A5), Бумага (130 г/м² мелованная), Цветность (4+4), Тираж (1000 шт), кнопка РАССЧИТАТЬ, поле Стоимость (2 500 руб (2.50 руб/шт)).
  • Оплата: способ (Ссылка на оплату и др.), кнопки ОТПРАВИТЬ ССЫЛКУ, ОТПРАВИТЬ СЧЁТ, ОПЛАЧЕНО.
  • История: хронология событий по сделке, например:
    • 14:30 | Система: Сделка создана из email
    • 14:35 | Менеджер: Расчёт выполнен, отправлена ссылка на оплату
  • Действия: ПЕРЕЙТИ К ПРОИЗВОДСТВУ, ЗАКРЫТЬ СДЕЛКУ, ОТМЕНА.

3.8. Упрощённый вариант для B2C (быстрый заказ)#

Для быстрых B2C-заказов (копирование, распечатка) — минимальная форма (пример — «Быстрый заказ (B2C) из email», статус «Новый»):

  • Клиент: Имя (Иван Иванов), Телефон (+7 999 123 45 67), Email (ivan@example.com).
  • Файлы: список с действием СКАЧАТЬ, например 📄 документ.pdf (1.2 MB).
  • Услуга: напр. Копирование A4, Кол-во, Сумма (пример: 20 шт = 100 руб).
  • Оплата: Сумма (100 руб), кнопка ОПЛАЧЕНО.
  • Действия: ВЫПОЛНИТЬ ЗАКАЗ, УВЕДОМИТЬ О ГОТОВНОСТИ, ЗАКРЫТЬ.

4. WhatsApp#

Назначение: один из текущих каналов переписки с клиентами. Включён в общую задачу объединения коммуникаций в CRM (наряду с Telegram и Макс), чтобы устранить дублирование информации по разным чатам.

⚠️ ОТКРЫТЫЙ ВОПРОС: Способ интеграции WhatsApp (официальный WhatsApp Business API, провайдер или иное) не определён. Не указано, является ли WhatsApp основным каналом.

5. ЭДО Диадок#

Назначение: юридически значимый электронный документооборот с юридическими лицами.

  • Электронное подписание договоров: некоторые B2B-клиенты просят составить договор — он подписывается электронно через ЭДО Диадок.
  • Отправка закрывающих документов (УПД) после выполнения заказа — также по ЭДО Диадок.

Интеграционная часть: CRM должна формировать и отправлять договоры и УПД через Диадок, отслеживая статус подписания.

⚠️ ОТКРЫТЫЙ ВОПРОС: Детали API Диадок (метод интеграции, объём автоматизации формирования и отправки документов) не специфицированы.

6. Бухгалтерия (Эльба, вопрос 1С)#

Назначение: выставление счетов и формирование закрывающих документов.

  • Эльба: экспорт заказов для счетов и УПД; синхронизация номеров счетов.
  • Способ: API Эльбы или экспорт/импорт XML.

⚠️ ОТКРЫТЫЙ ВОПРОС: Выбор бухгалтерской системы — Эльба или 1С (или иное). В разных частях источника фигурируют оба варианта; окончательный выбор не зафиксирован.

⚠️ ОТКРЫТЫЙ ВОПРОС: Доступен ли API Эльбы для нужных операций или интеграция будет реализована через файловый обмен XML.

7. Яндекс.Диск#

Назначение: хранение файлов заказов — папки заказов и синхронизация файлов (макетов). Способ: API Яндекс.Диска.

⚠️ ОТКРЫТЫЙ ВОПРОС: В разных частях проекта как хранилище фигурирует также S3 (для вложений email и макетов). Требуется согласовать единое решение по хранению файлов (Яндекс.Диск vs S3).

8. Курьерские службы (доставка)#

Назначение: создание заявок на доставку и получение трек-номеров.

Служба Что интегрируем Способ
Яндекс Доставка Создание заявок, получение трек-номера API Яндекс.Доставки
Достависта Создание заявок, получение трек-номера API Достависты
СДЭК Доставка API СДЭК

Контекст доставки: согласовываются дата самовывоза и пункт получения (три офиса на выбор) либо доставка с указанием времени, получателя, телефона. Курьерские заявки создаются для варианта доставки.

⚠️ ОТКРЫТЫЙ ВОПРОС: Финальный набор курьерских служб для интеграции (Яндекс / Достависта / СДЭК) не зафиксирован — перечислены как варианты.

9. Платежи (CloudPayments / ЮKassa / Т-Банк, 54-ФЗ)#

Назначение: онлайн-оплата заказов (преимущественно B2C), соблюдение 54-ФЗ (онлайн-кассы).

Рекомендуемые готовые модули для CRM:

  • CloudPayments (есть интеграция с Битрикс24).
  • Yandex.Kassa (ЮKassa) — много готовых решений.
  • Т-Банк (T-Pay, SberPay).

Эти сервисы уже поддерживают:

  • Онлайн-кассы (54-ФЗ).
  • Все способы оплаты.
  • Чековые ленты.
  • Возвраты.

Способы оплаты у клиента:

  • Физлицо: картой на сайте либо переводом на карту.
  • Юрлицо: по счёту (запрашивается ИНН для выставления счёта).
  • В email-сценарии B2C — ссылка на оплату или QR-код.

⚠️ ОТКРЫТЫЙ ВОПРОС: Конкретный платёжный провайдер (ЮKassa / CloudPayments / Т-Банк) не выбран. Онлайн-оплата отмечена как функциональность, внедряемая позже («позже»).

13. Система пропусков и контроля доступа#

Назначение: контроль доступа на охраняемый объект выдачи/доставки — офис на Орджоникидзе.

Полная механика пропусков (блок «Пропуск» в карточке заказа, QR-код для авто, SMS-код для пешего курьера, поля и кнопка «Запросить пропуск», AS-IS-процесс выдачи) описана в профильном документе — см. delivery.md, раздел «Управление пропусками и кодами доступа». Здесь фиксируется только интеграционный аспект.

Способ интеграции: кнопка «Запросить пропуск» вызывает API системы пропусков (если доступен) либо инициирует ручную заявку охране с заполнением данных в CRM. Пропуск выдаётся через сторонний сайт, наличие у которого API неизвестно.

⚠️ ОТКРЫТЫЙ ВОПРОС: Выдаются ли пропуска через какую-то систему (API) или всё вручную через заявку охране? Метод интеграции не подтверждён.
