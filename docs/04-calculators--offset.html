Калькулятор офсетной печати

Калькулятор офсетной печати реализует расчёт стоимости через партнёра (ГМ Принт): себестоимость партнёра берётся из матрицы цен, к ней применяется «чистая» (без налога) пропорциональная наценка, начисляется налог 7% и добавляется доставка. Документ описывает полный цикл, структуру БД, правила маржи, доставку, формулы расчёта, линейную интерполяцию маржи по сумме заказа, интерполяцию нестандартных размеров по площади и примеры расчётов.

Общая концепция#

Офсетная печать выполняется не на собственном оборудовании, а через партнёра (ГМ Принт). Партнёр предоставляет только себестоимость на стандартные форматы и тиражи. Система:

  1. Настройка (администратор):
    • Импорт цен партнёра в таблицу offset_partner_prices (только себестоимость).
    • Настройка правил маржи в таблице offset_margin_rules (по сумме или по тиражу).
    • Настройка налога (7%).
    • Настройка доставки (регионы, вес, тираж, временные интервалы).
  2. Клиент выбирает параметры: продукт (листовки, визитки, буклеты, брошюры, бирки, календари), формат (A4, A5, A6, A3, DL, 90×50), бумагу (130, 170, 300 г/м²), цветность (4+0, 4+4), тираж, дополнительные услуги (ламинация, биговка, фальцовка, сверление, скругление).
  3. Система рассчитывает стоимость: поиск себестоимости в offset_partner_prices, применение правил маржи (сначала по сумме, потом по тиражу), добавление стоимости доп. услуг, расчёт налога 7%, итоговая цена.
  4. Система предлагает доставку: бесплатная (10:00–18:00, без гарантии) — 0 руб; платная (до 13:00, с гарантией) — 300 руб; другие интервалы (утренняя, вечерняя); самовывоз со склада (суббота).
  5. Клиент оформляет заказ: выбирает дату/время доставки, оплачивает (ссылка/счёт), получает заказ.

Ключевые принципы:

  • Разделение данных: цены партнёра и правила маржи хранятся отдельно.
  • Маржа «чистая» — без налога, начисляется только на себестоимость партнёра.
  • Налог 7% добавляется поверх (себестоимость + маржа).
  • Поддержка сборных заказов (с цифровой и широкоформатной печатью).

Цены партнёра (только себестоимость)#

CREATE TABLE offset_partner_prices (
    id INT PRIMARY KEY AUTO_INCREMENT,
    product_type_id INT NOT NULL,
    format_id INT NOT NULL,
    paper_id INT NOT NULL,
    color_id INT NOT NULL,
    quantity INT NOT NULL,
    price DECIMAL(10,2) NOT NULL,
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (product_type_id) REFERENCES offset_product_types(id),
    FOREIGN KEY (format_id) REFERENCES offset_formats(id),
    FOREIGN KEY (paper_id) REFERENCES offset_papers(id),
    FOREIGN KEY (color_id) REFERENCES offset_colors(id),
    UNIQUE KEY unique_price (product_type_id, format_id, paper_id, color_id, quantity)
);

Цены на дополнительные услуги#

CREATE TABLE offset_service_prices (
    id INT PRIMARY KEY AUTO_INCREMENT,
    product_type_id INT NOT NULL,
    service_id INT NOT NULL,
    quantity_from INT,
    quantity_to INT,
    price DECIMAL(10,2) NOT NULL,
    is_active BOOLEAN DEFAULT true,
    FOREIGN KEY (product_type_id) REFERENCES offset_product_types(id),
    FOREIGN KEY (service_id) REFERENCES offset_services(id)
);

Правила маржи (наша наценка)#

CREATE TABLE offset_margin_rules (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(255) NOT NULL,
    -- Условия по себестоимости
    cost_from DECIMAL(10,2),
    cost_to DECIMAL(10,2),
    -- Условия по тиражу
    quantity_from INT,
    quantity_to INT,
    -- Ограничения по продукту (опционально)
    product_type_id INT,
    paper_id INT,
    color_id INT,
    -- Наценка
    margin_type ENUM('percent', 'fixed') NOT NULL,
    margin_value DECIMAL(10,2) NOT NULL,
    -- Приоритет (меньше = выше)
    priority INT DEFAULT 100,
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (product_type_id) REFERENCES offset_product_types(id),
    FOREIGN KEY (paper_id) REFERENCES offset_papers(id),
    FOREIGN KEY (color_id) REFERENCES offset_colors(id),
    INDEX idx_cost (cost_from, cost_to),
    INDEX idx_quantity (quantity_from, quantity_to)
);

Формулы расчёта#

Основная формула#

Цена_для_клиента = (Себестоимость + Наценка + Стоимость_услуг) × (1 + Налог/100) + Доставка

⚠️ ОТКРЫТЫЙ ВОПРОС: Порядок применения налога к доставке расходится между разделами источника. В разделе «Система офсетной печати» доставка добавляется ПОСЛЕ умножения на налог (доставка не облагается налогом). В сводной бизнес-логике (раздел 3.2) формула total = partner_price + delivery + tax + our_markup, где tax = partner_price × tax_percent / 100 начисляется только на себестоимость (без учёта наценки), а доставка прибавляется как отдельное слагаемое. Нужно зафиксировать единый порядок: облагается ли наценка налогом, и облагается ли доставка налогом.

Сводная формула из раздела бизнес-логики:

partner_price = getPriceFromMatrix(params)
delivery      = getDeliveryCost(city, weight)
tax           = partner_price × tax_percent / 100
our_markup    = partner_price × markup_percent / 100
total         = partner_price + delivery + tax + our_markup

Наценка (маржа)#

Правила применяются в порядке приоритета.

  1. По себестоимости:
    • 0–1000 руб → +1500 руб (фикс)
    • 1001–1500 руб → +1300 руб (фикс)
    • 1501–2500 руб → +60%
    • 2501–5000 руб → +50%
    • 5001+ руб → +40%
  2. Если нет правила по себестоимости — применяется правило по тиражу:
    • 1–100 шт → +1500 руб (фикс)
    • 101–500 шт → +1300 руб (фикс)
    • 501+ шт → +60%

Примечание: дискретные правила «по себестоимости/по тиражу» — это «старый» режим (calculation_method = 'discrete'). Для офсета рекомендуется плавная пропорциональная наценка (linear_percent, см. раздел «Плавная пропорциональная наценка»).

Налог#

Налог = (Себестоимость + Наценка + Услуги) × 7%

Налог не входит в правила маржи и применяется глобально для всех расчётов офсетной печати после вычисления price_before_tax. Ставку налога следует вынести в отдельную настройку (например, settings.tax_rate), изменяемую через админку.

Плавная пропорциональная наценка (linear_percent)#

Бизнес-требования#

  • Себестоимость заказа приходит от партнёра (ГМ Принт) в виде суммы partner_price (руб).
  • Маржа должна быть «чистой» — без налога, начисляется только на partner_price.
  • Налог 7% добавляется поверх итога (partner_price + маржа).
  • Цена для клиента должна меняться плавно при изменении partner_price, без резких скачков. Маржа не должна быть дискретной (разные проценты на интервалах с фиксированными границами), а должна быть непрерывной функцией от partner_price.
  • Для малых сумм нужна высокая маржа (чтобы не работать в убыток), для крупных — низкая (стимулировать объём).
  • Настройка должна быть простой для администратора: задать несколько опорных точек с процентами, система интерполирует остальное.

Мотивация нелинейности (исходные примеры заказчика):

partner_price маржа результат
1000 +100% 2000 руб
4000 +2000 руб (т.е. 50%) 6000 руб

При 4000 руб 100% дало бы 4000 руб маржи, но фактически нужно ~2000 (50%) — значит наценка нелинейная и процент снижается с ростом суммы.

Математическая модель#

Задаётся таблица опорных точек (price, margin_percent), где price — сумма заказа от партнёра, margin_percent — наценка в процентах от price для этой точки. Точки сортируются по возрастанию price. Между точками процент маржи изменяется линейно.

Для произвольной суммы S:

  • Находим две соседние точки (S1, P1) и (S2, P2), такие что S1 ≤ S ≤ S2.
  • Если S < S1 — берём P1.
  • Если S > S_last — берём P_last (фиксируем последний процент, без экстраполяции вверх).
percent(S)       = P1 + (P2 - P1) * (S - S1) / (S2 - S1)
margin           = S * (percent(S) / 100)
price_before_tax = S + margin
final_price      = price_before_tax * (1 + tax_rate)   // tax_rate = 0.07

Преимущества подхода#

  • Плавность — при увеличении заказа на 1 рубль итоговая цена меняется непрерывно, без «ступенек».
  • Пропорциональность — маржа считается как процент от суммы, но процент снижается с ростом объёма.
  • Простота настройки — администратору достаточно задать 5–7 точек, без интервалов и типов наценки.
  • Защита от убытков — для малых сумм можно установить высокий процент (100% и выше).
  • Стимулирование крупных заказов — большие суммы получают низкую маржу.
  • Масштабируемость — не нужно настраивать тысячи комбинаций (формат+бумага+тираж+цветность+допы), достаточно 5–10 точек.
  • Гибкость — при изменении цен партнёра достаточно скорректировать точки.

Интеграция в существующий модуль маржи#

Не ломая старую логику (для цифры и копирования), добавляется новый тип таблицы маржи.

В таблице margin_tables добавить поле calculation_method:

  • 'discrete' — старый режим (диапазоны с фикс или процентом) — для цифры и копирования.
  • 'linear_percent' — новый режим (интерполяция процента по точкам) — для офсета.

Для linear_percent поля quantity_from и quantity_to в их обычном смысле не используются. Источник описывает два варианта хранения опорных точек:

  • Вариант А — отдельная таблица margin_curve_points. Завести специализированную таблицу с полями point_price (сумма заказа от партнёра, руб) и margin_percent (процент маржи в этой точке). Семантически чище: точки кривой хранятся в собственной структуре, не перегружая поля margin_rules чужим смыслом.
  • Вариант Б (рекомендуется для унификации) — переиспользовать существующую margin_rules. Не вводить новых таблиц, а хранить точки в margin_rules, переинтерпретируя поля:
    • quantity_from = цена точки point_price (руб);
    • quantity_to = NULL;
    • margin_type всегда 'percent' (интерпретируется как процент именно для этой точки, а не для диапазона);
    • margin_value = процент маржи margin_percent;
    • сортировка по quantity_from.

В источнике поля точки прямо названы point_price (сумма) и margin_percent; в варианте Б они отображаются на quantity_from и margin_value соответственно. Дальнейшие разделы (псевдокод, указания программисту) описаны для варианта Б как основного, но при выборе варианта А логика интерполяции остаётся той же — меняется только источник точек.

⚠️ ОТКРЫТЫЙ ВОПРОС: В разделе про БД для офсета описана отдельная таблица offset_margin_rules (с полями cost_from/cost_to, quantity_from/quantity_to, margin_type, margin_value), а в разделе про плавную наценку предлагается переиспользовать общую таблицу margin_rules с полем calculation_method в margin_tables. Нужно решить, используется ли единая модель маржи для всех видов печати или отдельная таблица только для офсета.

Указания программисту (linear_percent)#

  • Расширить модель: добавить margin_tables.calculation_method (enum: 'discrete', 'linear_percent').
  • Выбрать способ хранения опорных точек (см. «Интеграция в существующий модуль маржи»): либо отдельная таблица margin_curve_points (поля point_price, margin_percent), либо переиспользование margin_rules (quantity_from = point_price, margin_value = margin_percent, margin_type = 'percent'). Для унификации с существующим модулем по умолчанию используется margin_rules.
  • В классе MarginCalculator создать метод _calculateLinearPercentMargin(partner_price), который: загружает точки для таблицы с calculation_method = 'linear_percent' (из выбранного хранилища — margin_curve_points или margin_rules), сортирует по сумме точки (point_price / quantity_from), находит отрезок, содержащий partner_price, вычисляет интерполированный процент, возвращает маржу в рублях.
  • Для 'discrete' оставить старую логику поиска по диапазонам.
  • В контроллере офсетного расчёта:
    1. Получить partner_price от API.
    2. Вызвать calculateMargin с total_price = partner_price и entity_type = 'offset'.
    3. Получить margin_amount.
    4. price_before_tax = partner_price + margin_amount
    5. final_price = price_before_tax * (1 + tax_rate)
  • Админка: при создании/редактировании таблицы маржи добавить выбор calculation_method. Для linear_percent — интерфейс ввода точек (цена, процент) с добавлением/удалением/сортировкой; для discrete — старый интерфейс с диапазонами.

Интерфейс настройки маржи с отображением чистой прибыли#

При настройке опорных точек администратор сразу видит чистую прибыль (в рублях) — это маржа, добавляемая к себестоимости партнёра до налога. Налог 7% на этом этапе не участвует.

Чистая прибыль = маржа в рублях = partner_price × (margin_percent / 100).

Промежуточные суммы (между точками) постоянно не показываются — но есть возможность проверить любую сумму по запросу.

Расчёт офсета для нестандартных размеров (интерполяция по площади)#

Задача#

Партнёр (ГМ Принт) предоставляет цены только на стандартные форматы. Клиент в калькуляторе может ввести произвольный размер (ширина × высота в мм). Для нестандартных размеров цены от партнёра нет, поэтому стоимость рассчитывается на основе цен ближайших стандартных форматов.

Требования и выбор подхода#

  • Если введённый размер точно соответствует стандартному формату — использовать цену этого формата.
  • Если размер нестандартный — два альтернативных подхода:
    1. Правило «вмещающий формат»: найти минимальный стандартный формат, полностью вмещающий заданный размер (ширина и высота не превышают размеры стандартного). Прозрачно для клиента (он платит как за больший формат). Возможен поворот макета — рекомендуется проверять оба варианта (ширина к ширине / высота к высоте и наоборот) и выбрать тот, который влезает; если ни один не влезает — взять следующий формат.
    2. Линейная интерполяция по площади (более гибкий): найти два ближайших стандартных формата (один меньше, другой больше по площади) и интерполировать цену.
  • Если размер превышает максимальный стандартный формат — заблокировать расчёт и предложить связаться с менеджером («Рассчитать у менеджера»).

⚠️ ОТКРЫТЫЙ ВОПРОС: Не зафиксирован единый метод для нестандартных размеров. Один источник рекомендует правило «вмещающий формат без поворота, иначе с поворотом, иначе ошибка» как более понятное клиенту; другой использует линейную интерполяцию по площади как основной метод. Также отмечено, что в офсете цена меняется скачком при переходе на другой формат (из-за раскладки на листе), что делает линейную интерполяцию потенциально неточной. Нужно выбрать один метод. Дополнительно: фактический метод расчёта на сайте ГМ Принт неизвестен (возможно, по большей стороне или сумме сторон, а не по площади) — требуется сверка.

Алгоритм интерполяции по площади#

  1. Получить список стандартных форматов с площадями и ценами (для каждого тиража, цвета, бумаги).
  2. Отсортировать форматы по возрастанию площади.
  3. Найти два опорных формата:
    • lower — формат с наибольшей площадью ≤ площади клиентского размера.
    • upper — формат с наименьшей площадью ≥ площади клиентского размера.
  4. Рассчитать цену:
    • Если найдены lower и upper:
      price = price_lower + (price_upper - price_lower) * (area_client - area_lower) / (area_upper - area_lower)
    • Если только upper (клиентский размер меньше минимального) — экстраполяция вниз:
      price = price_upper * (area_client / area_upper)
    • Если только lower (клиентский размер больше максимального) — вернуть ошибку (предпочтительно) либо экстраполировать вверх price = price_lower * (area_client / area_lower).
  5. Округлить цену до целых рублей.

area_client = clientWidth * clientHeight

Псевдокод#

function interpolatePrice(clientWidth, clientHeight, standardFormats) {
    const clientArea = clientWidth * clientHeight;
    // standardFormats: [{ width, height, price, area }]
    const sorted = [...standardFormats].sort((a, b) => a.area - b.area);

    let lower = null, upper = null;
    for (let f of sorted) {
        if (f.area <= clientArea) lower = f;
        if (f.area >= clientArea && !upper) upper = f;
    }

    if (lower && upper) {
        if (Math.abs(lower.area - upper.area) < 1e-6) return lower.price; // одинаковые площади
        const ratio = (clientArea - lower.area) / (upper.area - lower.area);
        return lower.price + (upper.price - lower.price) * ratio;
    } else if (upper && !lower) {
        // меньше минимального формата — экстраполяция вниз
        return upper.price * (clientArea / upper.area);
    } else if (lower && !upper) {
        // больше максимального — ошибка или экстраполяция
        throw new Error('Размер превышает максимальный стандартный формат');
    } else {
        throw new Error('Нет стандартных форматов');
    }
}

Полная формула с учётом тиража, цветности, бумаги#

Цены для каждого тиража, цветности и бумаги хранятся в матрице. Интерполяцию нужно проводить для каждой комбинации параметров отдельно.

  • Цветность и бумага: выбрать нужную подтаблицу цен.
  • Тираж: если цена партнёра задана для дискретных тиражей (например, 500, 1000, 2000) — сначала интерполировать по тиражу (для каждого формата), затем по полученным ценам выполнить интерполяцию по площади.
  • Раскладка на лист: в реальном офсете цена сильно зависит от того, сколько изделий помещается на печатный лист. Площадь — хороший прокси (она уже отражает изменение количества изделий на листе при переходе A7 → A6), но не идеальный; при необходимости можно ввести коэффициент использования листа.

Хранение цен: (format_id, quantity, color, paper) -> price.

Порядок при запросе клиента:

  1. Определить цветность и бумагу — выбрать подтаблицу.
  2. Для каждого формата получить цену, интерполированную по тиражу (если тираж не табличный).
  3. Вычислить площади всех форматов.
  4. Применить формулу интерполяции по площади.
  5. Округлить до целых рублей (или копеек).
  6. Выдать клиенту цену и сообщение: «Цена рассчитана на основе форматов A7 и A6».

Рекомендуется кэшировать результаты интерполяции для одинаковых размеров.

Что показать клиенту#

  • При интерполяции — сообщение: «Цена рассчитана на основе форматов A6 и A5» (с указанием фактических опорных форматов).
  • Если размер слишком большой — вместо цены кнопка «Рассчитать у менеджера».

Примеры расчётов интерполяции#

Пример 1. Размер 100×100 мм (площадь 10 000 мм²), интерполяция между A7 и A6.

Исходные данные (тираж недискретно указан; ранние цены):

  • A7 (74×105) → 7 770 мм² → 711 ₽
  • A6 (105×148) → 15 540 мм² → 1 460 ₽
  • A5 (148×210) → 31 080 мм² → 2 863 ₽ (для справки)
ΔP = 1 460 − 711 = 749 ₽
ΔS = 15 540 − 7 770 = 7 770 мм²
k  = (10 000 − 7 770) / 7 770 = 2 230 / 7 770 ≈ 0,2870
Цена = 711 + 749 × 0,2870 ≈ 711 + 214,9 = 925,9 ₽ → 926 ₽

Замечание: ранний ошибочный расчёт использовал пару A6/A5 и дал 960 ₽; правильнее использовать ближайший меньший формат A7 (площадь клиента 10 000 мм² лежит между A7 и A6), результат 926 ₽.

Пример 2. Размер 100×100 мм, тираж 10 000 шт.

Исходные данные (с сайта ГМ Принт):

  • A7 (74×105) → 7 770 мм² → 4 128 ₽
  • A6 (105×148) → 15 540 мм² → 7 642 ₽
ΔP = 7 642 − 4 128 = 3 514 ₽
ΔS = 15 540 − 7 770 = 7 770 мм²
k  = (10 000 − 7 770) / 7 770 = 2 230 / 7 770 ≈ 0,2870
Цена = 4 128 + 3 514 × 0,2870 = 4 128 + 1 008,5 ≈ 5 136,5 ₽ → 5 137 ₽

Пример 3. Размер 200×70 мм (площадь 14 000 мм²), тираж 10 000 шт.

Исходные данные (с сайта ГМ Принт, тираж 10 000):

  • A7 (74×105) → 7 770 мм² → 4 128 ₽
  • A6 (105×148) → 15 540 мм² → 7 642 ₽
  • A5 (148×210) → 31 080 мм² → 13 489 ₽ (для справки)

Площадь клиента 14 000 мм² лежит между A7 (7 770) и A6 (15 540), поэтому опорные форматы — A7 (нижний) и A6 (верхний).

S_lower = 7 770, price_lower = 4 128
S_upper = 15 540, price_upper = 7 642
S_c     = 14 000
ΔS = 15 540 − 7 770 = 7 770
ΔP = 7 642 − 4 128 = 3 514
k  = (14 000 − 7 770) / 7 770 = 6 230 / 7 770 ≈ 0,8018
Цена = 4 128 + 3 514 × 0,8018 = 4 128 + 2 817,5 ≈ 6 945,5 ₽ → 6 946 ₽
