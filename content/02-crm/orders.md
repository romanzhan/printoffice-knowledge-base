# Заказы: номер, статусы, карточка

Спецификация модуля заказов CRM PrintOffice: формат и правила генерации номера заказа, полная статусная модель B2B и упрощённая B2C, подстатусы производства, структура карточки заказа (все блоки), ручная корректировка цены с указанием причины и глобальный поиск заказов.

## Номер заказа

### Формат

Номер заказа — уникальный идентификатор с «расшифровкой» по составным частям:

```
ДДММГГ-Порядковый-Офис-Партнёр-Сборка
```

Пример: `290526-1-Л-Г-С`

| Часть | Расшифровка |
|-------|-------------|
| `290526` | Дата создания заказа (29 мая 2026) |
| `1` | Порядковый номер заказа (общий счётчик на день для всех офисов) |
| `Л` | Буква офиса (см. таблицу «Буквы офисов») |
| `Г` | Буква партнёра (см. таблицу «Буквы партнёров») |
| `С` | Сборный заказ: `С` — часть делаем мы, `_` — всё у одного исполнителя |

### Буквы офисов (пунктов)

| Буква | Офис / пункт |
|-------|--------------|
| П | Перово |
| В | Волгоградский |
| Л | Ленинский |
| Д | Доставка |

### Буквы партнёров

| Буква | Партнёр |
|-------|---------|
| Г | ГМ Принт |
| Э | Эдгар |
| ГО | Город (типография) |
| В | Виктория (типография Принтхит) |
| `_` | Нет партнёра |

> Коллизия буквы «В» (офис Волгоградский vs партнёр Виктория) — открытый вопрос, см. [11-open-questions.md](../11-open-questions.md#9-конфликт-буквы-в-в-системе-нумерации-заказов-офис-волгоградский-vs-партнёр-виктория).

### Правила генерации

- Дата и порядковый номер генерируются автоматически при создании заказа.
- Счётчик порядкового номера сбрасывается ежедневно.
- Порядковый номер — **общий счётчик на день** для всех офисов (а не отдельный для каждого офиса).
- Буква офиса выбирается менеджером либо проставляется автоматически по IP / адресу доставки.
- Буква партнёра и флаг `С` (сборка) проставляются автоматически после выбора технологии печати и добавления нескольких позиций с разными исполнителями.

### Поиск по номеру

Поиск по номеру заказа работает по любой части номера. Например, ввод `290526` покажет все заказы этого дня. (Подробнее о глобальном поиске — см. раздел «Глобальный поиск заказа».)

## Статусная модель

В системе используются две статусные модели: полная для B2B (сложные заказы) и упрощённая для B2C (быстрые заказы). Статусы визуально отображаются цветными метками в CRM.

### Полная модель B2B

Базовая последовательность статусов: Новая → Расчёт → Согласование → Оплата → В работе → Готово.

Полная статусная модель заказа (единая для B2B):

| Статус | Описание | Доступные действия |
|--------|----------|--------------------|
| Новый | Заказ создан (из сайта, email, менеджером) | Назначить ответственного, начать расчёт |
| На расчёте | Менеджер рассчитывает цену, уточняет параметры | Отправить КП, запросить макет |
| КП отправлено | Коммерческое предложение отправлено клиенту | Отслеживать просмотр, ждать согласования |
| Согласовано | Клиент утвердил КП | Выставить счёт, запросить оплату |
| Ожидает оплаты | Счёт отправлен, ждём платежа | Отметить оплаченным вручную или через платёжный шлюз |
| Оплачено | Деньги поступили | Передать в производство |
| В производстве | Заказ передан нашему производству или партнёру | Делится на подстатусы (см. ниже) |
| Готов к отгрузке | Изготовлен, ждёт передачи курьеру или клиенту | Сформировать отгрузочную накладную, вызвать курьера |
| Отгружен | Передан курьеру / клиенту | Отслеживать доставку, закрыть заказ |
| Завершён | Клиент получил, заказ закрыт | Сформировать УПД, отправить в Эльбу |
| Отменён | Заказ отменён (клиентом или менеджером) | Архивировать |

Машинные коды статусов B2B:

```javascript
const B2B_STATUSES = {
    'new':              'Новый',
    'calculation':      'Расчет',
    'proposal_sent':    'КП отправлено',
    'proposal_accepted':'Согласовано',
    'payment_pending':  'Ожидает оплаты',
    'paid':             'Оплачено',
    'production':       'В производстве',
    'ready':            'Готово',
    'delivery':         'Доставка',
    'completed':        'Завершен',
    'cancelled':        'Отменен'
};
```

⚠️ ОТКРЫТЫЙ ВОПРОС: Текстовый перечень статусов B2B содержит 11 статусов с отдельными «Готов к отгрузке» и «Отгружен», тогда как машинный перечень `B2B_STATUSES` объединяет их в `ready` («Готово») и `delivery` («Доставка»). Нужно зафиксировать единый канонический список статусов и их коды.

### Подстатусы производства

Для статуса «В производстве» (вкладка «Производство») предусмотрены подстатусы:

- Печать (наше производство / партнёр)
- Резка / Ламинация / Сборка
- Контроль качества

### Упрощённая модель B2C

Для B2C-заказов (быстрые заказы) применяется упрощённая статусная модель:

```javascript
const B2C_STATUSES = {
    'pending':    'В ожидании',   // клиент ещё не оплатил
    'processing': 'Выполняется',  // менеджер делает
    'ready':      'Готово',       // можно забирать
    'completed':  'Завершён',     // клиент забрал
    'cancelled':  'Отменён'
};
```

### Уведомления при смене статуса

- **Клиенту.** При смене любого статуса (кроме «Новый», «На расчёте», «КП отправлено») клиенту автоматически отправляется сообщение в мессенджер (Макс, Telegram) или email (канал настраивается). Клиент видит сообщения вида «Заказ в работе», «Заказ готов», «Заказ отгружен».
- **Менеджерам и сотрудникам.** Настраиваются в разделе «Уведомления»: можно выбрать статусы, канал (Макс, Telegram, email, всплывающее окно в CRM) и ответственных. Например: новый заказ → уведомление в общий чат и конкретному менеджеру.

## История изменений

Каждый заказ ведёт историю изменений (журнал событий). В блоке «История» карточки фиксируются действия системы, менеджера и клиента с датой, временем и автором события, например:

```
14.04.2026 10:30 | Система:         Заказ создан клиентом через сайт
14.04.2026 10:35 | Менеджер Анна:   Параметры проверены, расчёт выполнен
14.04.2026 10:40 | Система:         Отправлено КП клиенту (email, telegram)
14.04.2026 11:00 | Клиент:          Подтверждаю заказ
```

К записям истории можно добавлять комментарии менеджера (кнопка «Добавить комментарий»). Отдельно ведётся обоснование цен — фиксация причин изменения цены и их отправка клиенту (см. раздел «Обоснование цен» внутри «Корректировка цены»).

## Карточка заказа (B2B)

Полная карточка заказа состоит из набора блоков. Заголовок содержит номер заказа и текущий статус.

### Блок «Клиент»

- ФИО / название, телефон, email.
- Тип клиента (Физическое лицо / Юридическое лицо), признак постоянного клиента, действующая скидка (%).
- Действия: «Показать историю заказов», «Редактировать клиента».

### Блок «Состав заказа»

Перечень позиций заказа. Для каждой позиции отображаются:

- Наименование и технология (например, «Листовки А5 (цифровая печать)», «Контурная резка (широкоформат)»).
- Параметры (формат, бумага/материал, цветность, тираж/количество).
- Себестоимость, маржа (%), цена.
- Действие «Редактировать» (открывает расширенный режим редактирования позиции).

Под списком — кнопка «+ Добавить позицию».

Пример позиций:

| Позиция | Параметры | Себестоимость | Маржа | Цена |
|---------|-----------|---------------|-------|------|
| Листовки А5 (цифровая печать) | A5, 130 г/м² мелованная, 4+4, тираж 1000 шт | 1 400 руб | 60% | 2 240 руб |
| Контурная резка (широкоформат) | ПВХ 3 мм с контурной резкой, A3, 5 шт | 1 500 руб | 100% | 3 000 руб |

### Блок «Финансы»

Сводный финансовый итог по заказу:

- Себестоимость (сумма по позициям).
- Наша маржа (руб и %).
- Доставка (руб).
- ИТОГО.
- Статус оплаты (выпадающий список), действия «Отправить ссылку», «Отметить оплаченным».

Пример: Себестоимость 2 900 руб; Наша маржа 2 340 руб (81%); Доставка 0 руб; ИТОГО 5 240 руб.

### Блок «Доставка»

- Способ: Самовывоз / Доставка курьером / Доставка партнёром (выпадающий список).
- Адрес доставки.
- Дата, временной интервал, получатель.
- Действие «Рассчитать доставку».

### Блок «Файлы»

- Список прикреплённых файлов с именем и размером (например, `maket_final.pdf` 2.5 MB, `logo.png` 0.5 MB).
- Действия по файлу: «Скачать», «Предпросмотр», «Отправить клиенту».
- Действие «+ Загрузить файл».

### Блок «История»

Хронологический журнал событий заказа (см. раздел «История изменений»), действие «Добавить комментарий».

### Блок «Производство»

- Статус производства (выпадающий список, например «В работе»).
- Плановый срок.
- Этапы производства со статусами, например:
  - Этап 1: Печать (цифровая) — ВЫПОЛНЕНО
  - Этап 2: Резка (широкоформат) — В РАБОТЕ
  - Этап 3: Упаковка — ОЖИДАЕТ
- Действие «Перейти в производство».

### Действия по карточке

В нижней части карточки: «Сохранить», «Отправить клиенту», «Печать», «Удалить».

## Расширенный режим редактирования позиции

Открывается из блока «Состав заказа» по кнопке «Редактировать». Состоит из трёх блоков.

### Блок «Параметры»

- Тип продукции (выпадающий список).
- Технология (выпадающий список).
- Формат, ориентация.
- Бумага / материал.
- Цветность.
- Тираж (шт).
- Действие «Пересчитать».

### Блок «Расчёт»

- Себестоимость.
- Маржа (% — отмечается, стандартная или изменённая).
- Цена для клиента.
- Чекбокс «Ручная корректировка цены» с полем новой цены и обязательным полем «Причина» (например, «Скидка постоянному клиенту»).

Пример: Себестоимость 1 400 руб; Маржа 60% (стандартная); Цена для клиента 2 240 руб; ручная корректировка → 2 000 руб, причина «Скидка постоянному клиенту».

### Блок «Производство» (по позиции)

Производственные параметры позиции, рассчитываемые автоматически:

- Печатный лист и раскладка (например, SRA3, 8 шт/лист).
- Количество листов (например, 125).
- Время печати (например, 1.5 мин).
- Время резки (например, 7 мин).
- Действие «Пересчитать производство».

Действия блока: «Сохранить», «Удалить позицию», «Отмена».

## Корректировка цены

Цена позиции/заказа рассчитывается автоматически по правилам наценки, но может быть скорректирована менеджером вручную с **обязательным указанием причины**. Это обеспечивает прозрачность (менеджер видит, как считается цена) и контроль (любое отклонение от автоматического расчёта фиксируется с причиной).

### Блок «Корректировка цены» (на уровне заказа)

- **Автоматический расчёт** — исходная сумма (например, 5 240 руб).
- **Ручная корректировка** — редактируемые поля:
  - Маржа (%).
  - Доставка (руб).
  - Скидка (%).
  - Причина (текстовое поле, например «Постоянный клиент, срочная доставка»).
- **Результат** — итоговая сумма с указанием величины изменения относительно автоматического расчёта.
- Действия: «Применить», «Сбросить».

### Корректировка при расчёте позиции (что видит менеджер)

При расчёте стоимости позиции менеджер видит детализацию расчёта и блок корректировки.

Детализация расчёта показывает применённое правило наценки и итог. Пример для позиции «Листовки А4, 4+4, 130 г», тираж 27 шт:

```
Себестоимость за шт:        30 руб
Правило наценки:            диапазон 21-30 шт → 80%
Цена за шт для клиента:     54 руб
ИТОГО:                      1 458 руб
```

Блок корректировки (для менеджера) позволяет изменить цену за штуку и выбрать причину изменения:

- расчёт автоматический;
- скидка по согласованию;
- особая цена для постоянного клиента;
- ручной ввод (свободный текст, например «конкурентная цена»).

Пример результата корректировки: новая итоговая сумма 1 458 руб → 1 350 руб (скидка 108 руб). Действия: «Применить к позиции», «Отмена».

### Обоснование цен

Каждое изменение цены сопровождается обоснованием, которое фиксируется в истории заказа и может быть отправлено клиенту по выбранному каналу — **email, Telegram или WhatsApp**. В карточке заказа менеджер видит автоматически сгенерированную причину изменения (на основании применённого правила — например, изменение наценки при пересечении тиражного диапазона, срабатывание анти-скачка, применённая скидка постоянного клиента), может подставить готовый шаблон в один клик («Скидка постоянному клиенту», «Конкурентная цена», «Срочная доставка») и дополнить его произвольным ручным комментарием.

Это уровни 1–3 единой подсистемы обоснований (системные авто-причины → настраиваемые шаблоны → ручной текст; уровень 4 — комбинирование с автоматизацией ответов CRM). Полное описание 4 уровней автоматизации, каталог причин (`layout_optimization`/`make_ready`/`discount` и др.) с шаблонами сообщений и эталонный JSON-пример («листовки А5», −2800 руб / −46.3%) — см. [Автоматические обоснования изменения цены (4 уровня)](automations.md#автоматические-обоснования-изменения-цены-4-уровня).

### Анти-скачок (защита от резких цен)

На границах тиражных диапазонов цена за штуку падает, из-за чего итог за бо́льший тираж может оказаться ниже, чем за меньший. Механизм «анти-скачок» автоматически предлагает клиенту лучшую цену в таких случаях (например, «+1 экземпляр бесплатно»).

Что видит менеджер: при расчёте позиции, тираж которой близок к границе диапазона, в детализации появляется пометка о сработавшем анти-скачке и итог пересчитывается на лучшую цену (например, тираж 11 шт по 57 руб = 627 руб заменяется на 600 руб как за 10 шт). Причина изменения цены при этом формируется автоматически (см. раздел «Обоснование цен»).

Полное описание механики, псевдокод (`calculateWithRanges` / `useAntiJump`), таблица тиражей (5..21 шт при себестоимости 30 руб) и настройка в админке — в [модели ценообразования](../04-calculators/pricing-model.md#анти-скачок-активный-механизм-1-экземпляр-бесплатно).

## Упрощённая карточка заказа (B2C)

Для B2C используется упрощённая форма «Быстрый заказ», ориентированная на быстрое оформление и закрытие на месте.

### Блок «Клиент (минимально)»

Имя, Телефон, Email (опционально).

### Блок «Услуги (быстрый выбор)»

Кнопки быстрого добавления типовых услуг с фиксированной ценой:

| Услуга | Цена |
|--------|------|
| Копировать A4 | 5 руб |
| Копировать A3 | 10 руб |
| Распечатать A4 | 15 руб |
| Распечатать A3 | 25 руб |
| Фото | 50 руб |

Ниже — таблица добавленных услуг (Услуга, Кол-во, Цена, Сумма, действие удаления) и кнопка «+ Добавить услугу». Пример: Копирование A4 × 20 = 100 руб; Распечатка A4 × 5 = 75 руб.

### Блок «Оплата»

- ИТОГО (например, 175 руб).
- Способ оплаты: Наличные / Карта / Онлайн.

### Действия

«Выполнить и закрыть», «Только закрыть», «Отмена».

## Глобальный поиск заказа

Глобальная поисковая строка в CRM ищет заказ по любым данным и выдаёт результат за 1–2 секунды.

Поля поиска:

- Номер заказа (по любой части — см. раздел «Поиск по номеру»).
- Телефон.
- Email.
- Никнейм в Telegram.
- Адрес доставки.
- ИНН организации (для B2B).
- Название организации.
- Номер договора.
- Номер счёта.

Поиск должен показывать заказы, которые ещё не оплачены, но уже созданы, в том числе черновики и корзины.

⚠️ ОТКРЫТЫЙ ВОПРОС: Включение в результаты поиска неоплаченных заказов, черновиков и корзин зафиксировано в исходнике как открытый вопрос («Должен ли поиск показывать…?»). Требуется подтверждение состава отображаемых статусов в поисковой выдаче.
