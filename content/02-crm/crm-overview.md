# CRM: обзор

CRM — ядро системы PrintOffice (типография): единое рабочее пространство, в котором ведутся все заказы, клиенты, партнёры, доставка, файлы и коммуникации. Документ описывает назначение CRM, разделы главного меню, рабочий стол, типы заказов и их автоопределение, а также даёт обзор остальных под-документов CRM.

## Назначение CRM как ядра

CRM является центральным узлом системы и обеспечивает прозрачность работы для любого сотрудника, а не только для ведущего менеджера. Любой сотрудник (включая руководителя) должен иметь возможность открыть заказ и сразу понять его состояние.

### Прозрачность для любого сотрудника

Любой сотрудник, открыв заказ, должен сразу видеть:

- что печатаем;
- когда выдавать;
- кому выдавать;
- оплачен заказ или нет;
- где переписка с клиентом.

Решение — карточка заказа с вкладками:

- **«Основное»** — статус, ответственный, оплата.
- **«Состав»** — позиции заказа.
- **«Доставка / Отгрузка»** — когда, куда, кому.
- **«Коммуникации»** — история переписки из Telegram, WhatsApp, email, внутренние заметки.

Минимальный набор полей на первом экране карточки: номер заказа, клиент, статус, сумма, признак оплаты (оплачен / не оплачен), дата готовности, адрес отгрузки, перечень заказа.

История переписки отображается раздельно по каналам с возможностью переключения между ними.

### Идентификация заказа за 5 секунд

Любой сотрудник должен по звонку курьера или клиента за 5 секунд определить заказ:

- номер заказа;
- клиент (имя, телефон, адрес);
- что забирать (состав, количество мест);
- где забирать (адрес партнёра или нашего офиса);
- инструкции для курьера (пропуск, код, маршрут).

Поиск заказа возможен по нескольким идентификаторам: номеру заказа, телефону клиента, адресу доставки, ИНН организации, номеру счёта (через интеграцию с электронной бухгалтерией Эльба).

Номер заказа несёт «расшифровку» по схеме `ДДММГГ-Порядковый-Офис-Партнёр-Сборка` (например, `290526-1-Л-Г-С`): буква партнёра указывает исполнителя, буква **С** означает сборный заказ (часть делаем сами, часть — у партнёров). Это позволяет по номеру сразу понять, кто выполняет заказ и является ли он сборным. Полная спецификация формата, правила генерации и таблицы букв офисов и партнёров — в [orders.md](orders.md#номер-заказа).

## Совместная работа над заказом

Над одним заказом работают несколько человек: один принимает, считает, общается с клиентом; другой правит макеты и делает раскладку; третий передаёт в производство. Чтобы файлы не путались, используется интеграция с Яндекс.Диском с автоматически создаваемой структурой папок.

Пример структуры папок, создаваемой при создании заказа в CRM:

```text
/Заказы/2026/Май/Заказ_290526-1-Л/
    /01_Исходники_от_клиента/
    /02_Рабочие_макеты/        (общая папка)
    /03_Правки_Паша/           (личная папка сотрудника)
    /04_Готово_к_печати/       (финальные файлы)
```

Права доступа к папкам распределяются по ролям и стадиям заказа:

| Роль | Доступ |
|------|--------|
| Менеджер | На весь заказ |
| Дизайнер | На время правок |
| Печатник | На статусе «готово к печати» / «передано в печать» |
| Отгрузчик | На стадии отгрузки — папка «Готово к печати» |

Система прав должна быть гибкой и настраиваемой без программиста; по мере роста функциональности добавляются дополнительные возможности управления CRM. Требуется интеграция с Яндекс.Диском: открыть папку из карточки заказа, загрузить файл, посмотреть, кто и когда изменил файл, с сохранением истории изменений.

## Рабочие чаты внутри компании

В компании используется несколько рабочих чатов в мессенджере Макс:

- общий чат;
- чат производства;
- чат Пролетарской (запасное производство).

Проблема — дублирование информации и отсутствие связи с CRM. Решение — интеграция чатов в CRM: сообщения, относящиеся к конкретному заказу, автоматически попадают в карточку заказа как внутренние комментарии. Сообщения о смене статуса и любые вопросы по заказу попадают в карточку, чтобы любой сотрудник, не ведущий заказ, мог открыть карточку, понять историю и продолжить вести заказ.

Переписка разделяется на две непересекающиеся ветки:

- **внутренняя** — между сотрудниками по заказу;
- **с клиентом** — отдельная ветка.

Участники одной ветки не видят переписку из другой. Клиент не должен видеть переписку сотрудников. Предусматривается переключатель адресата сообщения, чтобы исключить ошибку отправки сообщения не тому получателю.

Основной канал коммуникаций — Макс, далее Telegram. Доступна отправка сообщений прямо из CRM, автоматическое создание заказа по сообщению (по настраиваемым правилам и условиям, можно отключить).

### Раздел бота

Предусмотрен настраиваемый раздел бота, который берёт на себя часть коммуникаций с клиентом:

- **Выяснение формальностей** — бот ведёт диалог, уточняя необходимые для оформления данные.
- **Парсинг данных в CRM** — телефон, адреса и прочие реквизиты, извлечённые из переписки, автоматически попадают в карточку клиента/заказа.
- **Авто-ответы на типовые вопросы** — график работы, адрес и как проехать, оплата, условия доставки, статусы заказа.

Правила работы бота настраиваются в разделе «Настройки» (справочник правил бота) без участия программиста.

⚠️ ОТКРЫТЫЙ ВОПРОС: способ организации веток переписки окончательно не выбран — рассматривается либо две отдельные непересекающиеся ветки, либо единая ветка с переключателем адресата и скрытием служебной переписки от клиента.

## Рабочий стол (дашборд)

Рабочий стол группирует заказы по статусам, отдельно отображая неоплаченные заказы. Используемые на рабочем столе статусы:

- **В работе** — заказы на согласовании, просчёте, подготовке макета, печати.
- **Ждёт отгрузки** — заказы, которые уже выполнены или выполняются, но не отгружены у партнёров; их нужно отгрузить в наш офис или клиенту.
- **Готовые к отгрузке**.
- **Выданные, но не оплаченные**.
- **Выданные и оплаченные**.

По всем заказам ведётся лента смены статусов в приложении Макс. Настраивается, по каким статусам приходит уведомление, в какой канал и какому менеджеру; уведомления можно отключать, в том числе внутри CRM. При переходе заказа в статус готовности к отгрузке по нему распечатывается отгрузочная накладная с QR-кодом (см. раздел «Доставка и отгрузка»).

Отдельная боль: сотрудники, принимая заказы, не всегда сообщают клиенту о сроках и статусе заказа. Поэтому требуется **автоматическое уведомление клиента при смене статусов** — клиент получает уведомление о ходе заказа без ручного вмешательства менеджера (в дополнение к уведомлениям менеджерам по ленте статусов).

## Партнёры (подрядчики)

CRM ведёт справочник партнёров-подрядчиков, у каждого из которых своя специализация, схема забора/отгрузки и характерные проблемы:

| Партнёр | Буква | Специализация | Логистика | Особенности |
|---------|-------|---------------|-----------|-------------|
| **ГМ Принт** | Г | Офсетная печать | Присылают своим курьером; иногда забираем сами при срочности | Всегда присылают письмо с номером своего заказа |
| **Эдгар** | Э | Широкоформатная печать | Курьеров не вызывает — всегда забираем сами | Путает / забывает отгрузить заказы: при большом числе заказов с разных точек может перепутать заказ, положить в отгрузку не тому курьеру — заказ уедет не туда |
| **Город** | ГО | Сложная продукция (браслеты, меню с лаком) | Забираем всегда сами | Всегда есть номер заказа, проблем с отгрузкой нет |
| **Виктория** (типография Принтхит) | В | Офсетная печать крупными тиражами | Забираем сами | Всегда есть номер заказа, проблем с отгрузкой нет. ⚠️ Буква «В» совпадает с буквой офиса Волгоградский — см. [11-open-questions.md](../11-open-questions.md#9-конфликт-буквы-в-в-системе-нумерации-заказов-офис-волгоградский-vs-партнёр-виктория) |

### Портал партнёра

Для партнёра (прежде всего Эдгар, при необходимости также Виктория и Город) предусмотрена отдельная страница/портал. На своей странице партнёр видит:

- номер заказа в работе;
- перечень работ и позиций в заказе;
- все свои заказы, которые он выполняет для нас, и от кого они поступили;
- конечный адрес отгрузки каждого заказа (а не только наш офис) — чтобы не путать заказы с разными адресами;
- назначенного на конкретный заказ курьера.

Партнёр может сам отмечать статус заказа на своей странице — мы видим эти изменения у себя. Альтернатива: если у партнёра есть собственная CRM, статус меняем мы сами по его уведомлению через свою CRM или мессенджер. Портал партнёра напрямую решает проблему Эдгара с путаницей при отгрузке.

## Доставка и отгрузка

Раздел **«Доставка»** — лента заказов в пути (от партнёров и наших курьерских заявок). В нём парсятся заказы на доставку из внешних источников:

- **ГМ Принт** присылает письмо с номером своего заказа, адресом доставки и ссылкой на маршрутизацию Яндекса, где указаны вес, адрес и время доставки;
- **Яндекс** и **Достависта** — наши курьерские заявки (могут звонить как менеджеру, так и руководителю, если не дозвонились менеджеру).

Адрес доставки всегда называет клиент; его заносит в базу CRM менеджер или сам клиент (адрес привязан к карточке клиента и его данным).

### Блок «Отгрузка» в карточке заказа

Блок отгрузки заполняется частично автоматически: данные берутся с сайта при оформлении заказа клиентом либо вносятся менеджером по продажам (позиции, адреса, контакты клиента). При оформлении доставки заказ в нашей системе связывается с системой Достависты или Яндекса.

### Отгрузочная накладная

При готовности заказа к отгрузке распечатывается отгрузочная накладная, содержащая:

- QR-код;
- номер заказа;
- контакты клиента;
- адрес;
- перечень позиций с количеством.

Печать накладной с QR-кодом привязана к моменту готовности заказа к отгрузке (лента смены статусов при этом идёт в приложении Макс).

### Пропуска и коды доступа

Для офиса на Орджоникидзе при отгрузке оформляется пропуск (авто → QR-код, пеший курьер → SMS-код); в карточке заказа при выборе этого адреса появляется блок «Пропуск». Полное описание механики — см. [«Доставка и отгрузка», раздел «Управление пропусками и кодами доступа»](../07-partners-delivery/delivery.md#6-управление-пропусками-и-кодами-доступа).

## Типы заказов и их автоопределение

CRM автоматически определяет тип заказа на входе на основе параметров (тип клиента, набор услуг, тираж, потребность в дизайне, срочность, способ оплаты, способ получения).

### Категории заказов

| Тип | Примеры продукции | Процесс | Закрытие |
|-----|-------------------|---------|----------|
| **B2C (быстрый)** | Копирование, распечатка, фото, срочная полиграфия | Упрощённый: минимум полей, оплата на месте, выдача сразу, статус `completed` | Чек на email/телефон |
| **B2C (обычный)** | Листовки (малый тираж), визитки, наклейки | Полный: все поля, оплата онлайн/счёт, доставка/самовывоз, полная цепочка статусов | Чек |
| **B2B (сложный)** | Крупные тиражи, брошюры, дизайн, каталоги | Расширенный: все поля + договор, счёт, УПД, ЭДО, доставка | УПД, акт, закрытие |

### Цепочки статусов

```javascript
// B2C (быстрые заказы)
pending → processing → ready → completed

// B2B (сложные заказы)
new → calculation → proposal_sent → proposal_accepted
    → payment_pending → paid → production → ready → delivery → completed
```

Полный жизненный цикл заказа в карточке: `новый → расчёт → КП → согласование → оплата → производство → готов → выдан`.

### Алгоритм определения типа заказа

Тип заказа определяется по входным параметрам: `clientType` (`b2c`/`b2b`), `services`, `quantity`, `hasDesign`, `isUrgent`, `paymentMethod` (`cash`/`card`/`invoice`), `pickupMethod` (`office`/`delivery`).

```javascript
class OrderTypeDetector {
    /**
     * Определение типа заказа на основе параметров
     */
    detectOrderType(params) {
        const {
            clientType,     // 'b2c' / 'b2b'
            services,       // список услуг
            quantity,       // тираж
            hasDesign,      // нужен ли дизайн
            isUrgent,       // срочный?
            paymentMethod,  // 'cash', 'card', 'invoice'
            pickupMethod    // 'office', 'delivery'
        } = params;

        // Признаки быстрого B2C заказа
        const isFastB2C = (
            clientType === 'b2c' &&
            paymentMethod !== 'invoice' &&
            pickupMethod === 'office' &&
            !hasDesign &&
            (services.includes('copy') || services.includes('print')) &&
            quantity < 100
        );

        // Признаки сложного B2B заказа
        const isComplexB2B = (
            clientType === 'b2b' ||
            paymentMethod === 'invoice' ||
            pickupMethod === 'delivery' ||
            hasDesign ||
            quantity > 500 ||
            services.includes('brochure')
        );

        if (isFastB2C && !isComplexB2B) {
            return {
                type: 'fast',
                statusFlow: B2C_STATUSES,
                requiresContract: false,
                requiresInvoice: false,
                autoComplete: true
            };
        }

        if (isComplexB2B) {
            return {
                type: 'complex',
                statusFlow: B2B_STATUSES,
                requiresContract: true,
                requiresInvoice: true,
                autoComplete: false
            };
        }

        return {
            type: 'regular',
            statusFlow: B2B_STATUSES,
            requiresContract: false,
            requiresInvoice: false,
            autoComplete: false
        };
    }
}
```

Сводка по результату определения:

| Тип (`type`) | `statusFlow` | `requiresContract` | `requiresInvoice` | `autoComplete` |
|--------------|--------------|--------------------|--------------------|----------------|
| `fast` | `B2C_STATUSES` | false | false | true |
| `complex` | `B2B_STATUSES` | true | true | false |
| `regular` | `B2B_STATUSES` | false | false | false |

⚠️ ОТКРЫТЫЙ ВОПРОС: пороги тиража в алгоритме (`quantity < 100` для быстрого B2C и `quantity > 500` для сложного B2B) оставляют «серую зону» 100–500 шт., которая попадает в тип `regular` только при отсутствии других признаков сложности; конкретные граничные значения требуют подтверждения.

## Структура модуля CRM (под-документы)

Модуль CRM состоит из следующих блоков, детализируемых в отдельных под-документах:

- **Заказы** — жизненный цикл (новый → расчёт → КП → согласование → оплата → производство → готов → выдан); карточка заказа (позиции, статусы, история); ручная корректировка цены, маржи, доставки, скидки; коммерческие предложения (генерация, обоснование цен); сборные заказы (цифра + офсет + широкоформат).
- **Клиенты** — типы B2C (физ. лица) и B2B (юр. лица); регистрация по email, телефону, Telegram, соцсетям; вход по магической ссылке (без пароля) и по паролю (для сотрудников); личный кабинет (история заказов, повтор заказов, брифы); сегментация (постоянные, новые, VIP).
