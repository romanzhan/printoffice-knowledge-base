# Автоматизации и cron

Описание автоматических процессов CRM PrintOffice: автосоздание производственных заданий и папок при оформлении заказа, правила-триггеры (автодействия и автоуведомления при событиях), а также фоновые cron-задачи (напоминания о неоплаченных заказах, автоматическая чистка и архивация файлов).

## Автоматическое создание производственного задания при оформлении заказа

Триггер «при создании заказа → задание партнёру»: при оформлении заказа клиентом система автоматически решает, формировать ли задание для партнёра-подрядчика, и маршрутизирует его по типу печати (`printing_type`) — `offset` → ГМ Принт (`partner_id = 1`), `wide` → партнёр по широкоформату (`partner_id = 2`). Задание создаётся со статусом `new`, в него собирается спецификация заказа и прикрепляются файлы, после чего партнёру уходит уведомление.

Полный код класса `AutoTaskCreator`, спецификация задания и описание таблицы `partner_tasks` вынесены в профильный модуль партнёров: см. [Партнёры и подрядчики → Автоматическое создание задания](../07-partners-delivery/partners.md#52-автоматическое-создание-задания) и [таблицу `partner_tasks`](../07-partners-delivery/partners.md#62-таблица-partner_tasks-задания-для-партнёров).

⚠️ ОТКРЫТЫЙ ВОПРОС: в исходных материалах раздел назван «Автоматическое создание производственного задания/папок», однако реализация описывает только создание записи задания (`partner_tasks`) и привязку файлов. Отдельный механизм создания файловых «папок» (структуры каталогов в хранилище) для заказа/задания не специфицирован — требуется уточнить, что подразумевается под «папками».

## Правила автоматизации (триггеры и действия)

Система автоматизации в CRM построена на правилах вида «КОГДА (триггер) → СДЕЛАТЬ (действие)». Правила хранятся в таблице `automation_rules`, имеют приоритет (правило с большим приоритетом применяется первым) и могут быть включены/выключены.

### Таблица правил автоматизации

```sql
CREATE TABLE automation_rules (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(255),
    is_active BOOLEAN DEFAULT true,

    -- Триггер (когда срабатывает)
    trigger_type VARCHAR(50),       -- 'price_change', 'status_change', 'manager_action'
    trigger_conditions JSON,        -- {"price_change_percent": ">20", "has_lamination": true}

    -- Действие (что сделать)
    action_type VARCHAR(50),        -- 'send_message', 'assign_manager', 'create_task'
    action_config JSON,             -- {"channel": "telegram", "template_id": 5}

    -- Приоритет (какое правило применять первым)
    priority INT DEFAULT 0,

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### Типы триггеров

| trigger_type | Событие |
|---|---|
| `price_change` | Изменение цены заказа/позиции |
| `status_change` | Смена статуса заказа |
| `manager_action` | Действие менеджера |

### Типы действий

| action_type | Действие |
|---|---|
| `send_message` | Отправить сообщение клиенту (канал и шаблон в `action_config`) |
| `assign_manager` | Назначить менеджера |
| `create_task` | Создать задачу (приоритет, исполнитель, срок в `action_config`) |

Дополнительно действие может включать отправку уведомления менеджеру в Telegram и приостановку автоотправки клиенту (опционально).

### Примеры правил

```sql
-- Правило 1: Большое снижение цены → создать задачу старшему менеджеру
INSERT INTO automation_rules VALUES (
    1,
    'Большое снижение цены - уведомить менеджера',
    true,
    'price_change',
    '{"percent": "<-30", "reason": "layout_change"}',
    'create_task',
    '{"priority": "high", "assign_to": "senior_manager", "due_in_hours": 1}',
    100
);

-- Правило 2: Стандартное изменение цены → автоответ клиенту
INSERT INTO automation_rules VALUES (
    2,
    'Стандартное изменение - автоответ клиенту',
    true,
    'price_change',
    '{"percent": "between:-15 and 15"}',
    'send_message',
    '{"channel": "telegram", "template": "price_adjusted_standard", "auto_approve": true}',
    50
);
```

Семантика условий триггера:
- `"percent": "<-30"` — снижение цены более чем на 30%.
- `"percent": "between:-15 and 15"` — изменение цены в коридоре от −15% до +15% (отправляется автоответ с автоутверждением).

### Пример настройки правила (интерфейс)

Правило «Большое снижение цены»:
- КОГДА: изменение цены > 30% в минус И причина — изменение раскладки.
- СДЕЛАТЬ:
  - Создать задачу менеджеру (приоритет высокий, исполнитель — старший менеджер, срок 1 час).
  - Отправить уведомление в Telegram менеджеру (`@manager_chief`) с текстом «Крупное изменение цены по заказу...».
  - (Опционально) Приостановить отправку клиенту.

## Автоматические обоснования изменения цены (4 уровня)

Подсистема обоснований работает на четырёх уровнях автоматизации; правила автоматизации (`automation_rules`) — это уровень 4.

| Уровень | Описание |
|---|---|
| 1 | Системные обоснования — генерируются автоматически по правилам |
| 2 | Шаблонные обоснования — настраиваются без программиста |
| 3 | Ручные обоснования — менеджер пишет свободный текст |
| 4 | Комбинированные (авто + ручные правки) / автоматизация ответов в CRM |

### Уровень 1: автоматические системные обоснования

Система анализирует, почему изменилась цена, и генерирует объяснение. Каталог типовых причин (с кодами и шаблонами сообщений):

| Код | Причина | Категория | Пример сообщения |
|---|---|---|---|
| `layout_optimization` | Оптимизация раскладки | Технологическая | «Раскладка изменена с {oldLayout} шт/лист на {newLayout} шт/лист для совместимости с {operation}. Это увеличило количество листов с {oldSheets} до {newSheets}.» |
| `material_optimization` | Оптимизация материалов | Материальная | «Для {material} выбран оптимальный формат {newFormat} вместо {oldFormat} (экономия {saved} руб)» |
| `operation_required` | Добавлена технологическая операция | Операция | «Для качественного выполнения заказа требуется {operation}. Это добавило {cost} руб к стоимости.» |
| `complexity_increased` | Повышенная сложность | Сложность | «Учтена повышенная сложность: {reason}. Коэффициент сложности {factor}.» |
| `urgency` | Срочность выполнения | Наценка | «Срочное выполнение за {hours} часов. Коэффициент срочности {factor}.» |
| `make_ready` | Приладка оборудования | Наценка | «Сложная настройка оборудования (приладка): {cost} руб.» |
| `discount` | Скидка | Скидка | «Скидка {percent}% на позицию "{itemName}" (экономия {saved} руб)» или «Скидка {percent}% на весь заказ (экономия {saved} руб)» |

Алгоритм генерации (`PriceJustificationGenerator`):
- Сравнивает исходный и скорректированный расчёты, вычисляет суммарное изменение и процент:
  - `totalChange = adjusted.total - original.total`
  - `percentChange = (adjusted.total - original.total) / original.total * 100`
- Анализирует каждое изменение, группирует причины на повысившие (`increased`) и понизившие (`decreased`) цену.
- Формирует итоговое резюме: при росте — «Цена увеличилась на {N} руб ({M}%) из-за: ...»; при снижении — «Цена снизилась на {N} руб ({M}%) благодаря: ...»; при отсутствии изменения — «Цена не изменилась».

### Уровень 2: настраиваемые шаблоны обоснований

Шаблоны хранятся в БД и редактируются без программиста. Поддерживаются переменные с подстановкой значений и условия автоматического применения:

| Переменная | Значение |
|---|---|
| `{{operation}}` | Название операции |
| `{{action}}` | «повысило» / «изменило» (направление влияния на цену) |
| `{{amount}}` | Сумма изменения |
| `{{percent}}` | Процент изменения |
| `{{old_layout}}` / `{{new_layout}}` | Исходная и новая раскладка (для шаблонов раскладки) |
| `{{reason}}` | Причина изменения |

Примеры текста шаблонов (редактируются в визуальном редакторе шаблонов):
- Технологическая операция: «Для качественного выполнения заказа требуется {{operation}}. Это {{action}} стоимость на {{amount}} руб.»
- Сложная раскладка: «Из-за нестандартной раскладки ({{old_layout}}→{{new_layout}}) цена скорректирована на {{amount}} руб».
- Срочность: «Срочное выполнение за {{hours}} ч (коэф. {{factor}})».

```sql
CREATE TABLE justification_templates (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(255),                  -- Название шаблона
    code VARCHAR(100),                  -- Уникальный код
    category VARCHAR(50),               -- 'price_increase', 'price_decrease', 'info'

    template_text TEXT,                 -- "Цена {action} на {amount} руб из-за {reason}"
    available_variables JSON,           -- ["action", "amount", "reason", "percent"]
    applies_to VARCHAR(50),             -- 'order', 'item', 'both'
    auto_apply_conditions JSON,         -- {"price_change_percent": ">10", "has_operation": "lamination"}

    is_active BOOLEAN DEFAULT true,
    sort_order INT DEFAULT 0,
    created_by INT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);

CREATE TABLE template_variables (
    id INT PRIMARY KEY AUTO_INCREMENT,
    template_id INT,
    variable_name VARCHAR(100),
    variable_type VARCHAR(50),          -- 'text', 'number', 'currency', 'percent'
    default_value VARCHAR(255),
    description TEXT,
    FOREIGN KEY (template_id) REFERENCES justification_templates(id)
);
```

Условие автоматического применения шаблона может быть глобальным («при добавлении любой операции») или ограниченным конкретными операциями (например, ламинация, резка).

### Уровни 3–4: ручные правки и комбинирование

Менеджеру предоставляется сводный интерфейс: автоматически сгенерированные причины (с возможностью удалить/отредактировать) плюс ручное добавление обоснования (через шаблон или свободный текст с указанием влияния на цену). Итоговый текст для клиента собирается из авто- и ручных частей. Около 80% обоснования генерируется автоматически, остальное менеджер дополняет вручную; затем выбирается канал отправки.

Пример интерфейса обоснования (заказ «Листовки А5, 100 шт», изменение −2400 руб / −39.7%):
- Блок «Автоматически сгенерированные причины» — изменение раскладки (24 → 20 шт/лист для совместимости с ламинацией) и добавленная приладка (+500 руб), каждая причина с кнопками «Удалить»/«Редактировать».
- Блок «Добавить обоснование вручную» — выбор шаблона или свободный текст; в примере менеджер вводит «Для постоянного клиента предоставлена дополнительная скидка 10%» и указывает влияние на цену −350 руб (скидка).
- Блок «Итоговый текст для клиента» — персонализированное письмо (обращение «Уважаемый Иван Петрович!»), перечень причин изменения, новая стоимость 3250 руб и ссылка для согласования; завершается кнопкой «Отправить клиенту».

В результате клиент получает:
- Прозрачное объяснение, за что и почему изменилась цена.
- Возможность согласовать предложение в 1 клик (по ссылке согласования).
- Доверие к типографии за счёт открытости расчёта.

### Преимущества системы обоснований

- Прозрачность — клиент понимает, за что платит.
- Автоматизация — около 80% обоснований генерируются автоматически.
- Гибкость — менеджер может добавить ручной текст.
- Настраиваемость — шаблоны редактируются без программиста.
- Контроль — система следит за крупными изменениями (через правила-триггеры уровня 4).
- История — все обоснования сохраняются.

### Пример работы (листовки А5 с ламинацией)

Клиент заказывает 100 шт А5, 4+4, ламинация; автоматический расчёт — 6050 руб. При проверке выявлена несовместимость раскладки с ламинацией, цена пересчитана:

```json
{
  "originalPrice": 6050,
  "newPrice": 3250,
  "change": -2800,
  "percent": -46.3,
  "reasons": [
    {
      "type": "layout_changed",
      "title": "Изменение раскладки",
      "message": "Раскладка изменена с 24 шт/лист на 20 шт/лист для совместимости с ламинацией. Количество листов увеличено, но себестоимость снижена за счёт оптимизации.",
      "impact": -2500
    },
    {
      "type": "make_ready_added",
      "title": "Приладка",
      "message": "Добавлена приладка для точной настройки оборудования: +500 руб",
      "impact": 500
    },
    {
      "type": "discount",
      "title": "Скидка",
      "message": "Для постоянного клиента скидка 15% на позицию",
      "impact": -800
    }
  ]
}
```

⚠️ ОТКРЫТЫЙ ВОПРОС: в текстовом примере резюме указано снижение −2400 руб (−39.7%) и итоговая стоимость 3250 руб, тогда как в JSON-примере того же сценария указано изменение −2800 руб (−46.3%). Числа в двух местах не согласованы — требуется зафиксировать единый корректный расчёт.

## Автонапоминания о неоплаченных заказах

Для отслеживания неоплаченных заказов система автоматически напоминает клиенту о необходимости оплаты по расписанию.

| Параметр | Значение |
|---|---|
| Триггер | Заказ не оплачен |
| Напоминания | Через 1, 3 и 7 дней после создания/выставления счёта |

Связанные процессы по платежам и закрытию заказа:
- При оплате по счёту на этапе закрытия заказа создаётся УПД (универсальный передаточный документ) и отправляется по ЭДО (Эльба).
- Постоянным клиентам доступна отсрочка платежа (отметка «постоянный»).

⚠️ ОТКРЫТЫЙ ВОПРОС: не определено, от какой именно даты отсчитываются интервалы 1/3/7 дней (создание заказа, выставление счёта или иной момент), и каков канал/шаблон напоминания. Требуется уточнить точку отсчёта и механику отправки.

## Cron-задачи: автоматическая чистка и архивация файлов

Файлы классифицируются по категориям (`temp`, `active`, `archive`, `trash`) и имеют срок хранения (`expires_at`). Фоновый сервис `FileCleanupService` периодически удаляет, архивирует и дедуплицирует файлы.

### Сроки хранения по умолчанию

| Категория / контекст | Срок жизни |
|---|---|
| Временные файлы (`temp`) | 72 часа (3 дня) |
| Активные файлы (`active`) после завершения заказа | 30 дней |
| Файлы в корзине (`trash`) | 30 дней до окончательного удаления |
| Дубликаты, отправленные в корзину | 7 дней |

Срок жизни временных файлов задаётся в конфигурации `FileManager` (`tempFileLifetime = 72 * 60 * 60 * 1000` — 72 часа / 3 дня); активные файлы — 30 дней после завершения заказа (`activeFileLifetime`), корзина — 30 дней (`trashLifetime`). Полная итоговая таблица сроков (с учётом типа клиента — пробная загрузка, B2C, B2B, постоянный макет) приведена в материале по файловому хранилищу.

### Расписание cron-задач

| Задача | Периодичность | Cron-выражение | Действие |
|---|---|---|---|
| `cleanupTempFiles` | Каждый час | `0 * * * *` | Окончательно удаляет временные файлы с истёкшим `expires_at` |
| `runAllCleanups` (полная очистка) | Ежедневно в 03:00 | `0 3 * * *` | Запускает все задачи очистки подряд |
| `cleanupActiveFiles` | Ежедневно (в составе полной очистки) | — | Переносит в архив активные файлы завершённых заказов старше 30 дней |
| `cleanupTrash` | Ежедневно (в составе полной очистки) | — | Окончательно удаляет файлы из корзины с истёкшим сроком |
| `deduplicateFiles` | Ежедневно (в составе полной очистки) | — | Находит дубликаты по хешу файла, оставляет один, остальные → корзина на 7 дней |

Используется библиотека `node-cron`.

### Реализация

Полный код класса `FileCleanupService` (методы `cleanupTempFiles`, `cleanupActiveFiles`, `cleanupTrash`, `deduplicateFiles`, `runAllCleanups`) и регистрация cron-задач (`node-cron`, расписания `0 * * * *` и `0 3 * * *`) приведены в профильном документе по файлам: [Управление файлами и макетами → Автоматическая чистка файлов (cron-задачи)](../05-prepress/file-management.md#4-автоматическая-чистка-файлов-cron-задачи). Здесь дублирование кода намеренно не приводится.

⚠️ ОТКРЫТЫЙ ВОПРОС: в исходных материалах указан как cron-механизм автоматическая чистка файлов, но напоминания о неоплаченных заказах и проверка email/платежей описаны лишь как требования без cron-расписания. Точный набор cron-задач по платежам/почте не специфицирован — требуется определить периодичность и реализацию проверки оплаты и входящей почты.
