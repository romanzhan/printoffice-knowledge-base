# Партнёры и подрядчики

Профили партнёров и подрядчиков типографии PrintOffice (офсет, широкоформат, сувенирка), правила отгрузки, а также модуль «Партнёры» с личным кабинетом партнёра (токен-ссылка, смена цены/срока с историей, статусы, видимость адреса доставки) и принципы прозрачности загрузки.

## 1. Контекст и список партнёров

Часть продукции PrintOffice выполняет силами подрядчиков — собственного производства широкоформатной, офсетной и сувенирной печати у компании нет.

Активные подрядчики по чек-листу подготовки к встрече с архитектором: **GMPrint, Wideformat, ФастПринт, ПринтХит** (`raw/architecture.md:1070`). Развёрнутые профили партнёров и их специализация даны заказчиком отдельно (`raw/crm-site-setup.md:180–185`) и приведены ниже — при расхождении верить развёрнутым профилям.

Основные партнёры и подрядчики (по развёрнутым профилям заказчика):

| Партнёр / подрядчик | Тип печати / продукции | Как работаем |
| --- | --- | --- |
| ГМ Принт (GMPrint) | Офсетная печать | Отправляют заказ своим курьером; иногда забираем сами, если срочно. Офсетный партнёр с собственным сайтом и прайсом (`partner_id = 1` в автоматике). |
| Эдгар | **Широкоформатная печать** | **Всегда забираем сами.** Известная проблема: при большом потоке заказов с разных точек Эдгар может перепутать/забыть отгрузку — поэтому отгрузку от Эдгара контролируем особо. Для него в первую очередь делается портал/ЛК партнёра. |
| Типография «Город» | **Сложная продукция: печать браслетов, меню с лаком** | Забираем всегда сами; всегда есть номер заказа, проблем с отгрузкой нет. Интеграция без портала — статусы ведёт менеджер (возможна и страница партнёра). |
| Виктория / Принтхит (PrintHit) | **Офсетная печать, очень крупные тиражи** | Забираем сами; всегда есть номер заказа, проблем с отгрузкой нет. **«Виктория» и «Принтхит» — одна сущность** (типография Принтхит). |
| FastPrint (Фастпринт) | (специализация в первоисточнике не раскрыта) | Указан в чек-листе активных подрядчиков; развёрнутого профиля/прайса/способа отгрузки в материалах нет. |
| Подрядчики сувенирки | Сувенирная продукция | Перезаказ через мессенджер (Telegram), ручной запрос цены и срока. |

> **Правка аудита (Этап 1):** ранее в этой таблице Виктория/Принтхит ошибочно числилась широкоформатным подрядчиком, а профиль Эдгара значился «не раскрыт». Исправлено по явным формулировкам заказчика (`raw/crm-site-setup.md:181–183`): Эдгар = широкоформат, Виктория/Принтхит = офсет (крупный тираж), Город = сложная продукция.

⚠️ ОТКРЫТЫЙ ВОПРОС: По AS-IS офсетных партнёров заявлено **три** (`raw/architecture.md:1076`), но по развёрнутым профилям явно офсетные только два — ГМ Принт и Виктория/Принтхит. Кто третий офсетный партнёр (возможно FastPrint, чей профиль не раскрыт) — требуется уточнить. Связано с коллизией буквы «В» в номере заказа (офис Волгоградский vs партнёр Виктория) — см. [11-open-questions.md](../11-open-questions.md#9-конфликт-буквы-в-в-системе-нумерации-заказов-офис-волгоградский-vs-партнёр-виктория).

⚠️ ОТКРЫТЫЙ ВОПРОС: В автоматике (`AutoTaskCreator`) широкоформат маршрутизируется на `partner_id = 2`. Нужно явно зафиксировать, какой партнёр соответствует `partner_id = 2` для широкоформата (по профилям это **Эдгар**), и свести справочник `partner_id ↔ партнёр ↔ тип печати` воедино.

## 2. Профили подрядчиков по типам печати

### 2.1. Офсетная печать

- У офсетных партнёров есть собственные сайты с выставленными ценами и сроками.
- Менеджер списывает цены с сайта партнёра и добавляет наценку с учётом расходов: налоги и доставка до офиса PrintOffice либо до клиента.
- По AS-IS офсетных партнёров заявлено **3**, у каждого свои цены и сроки (`raw/architecture.md:1076`). По развёрнутым профилям явно офсетные — **ГМ Принт** и **Виктория/Принтхит** (крупные тиражи); третий не идентифицирован (см. ⚠️ в разделе 1).
- ГМ Принт: цены и сроки — с собственного сайта; отправляет заказ своим курьером, иногда забираем сами при срочности.
- Виктория/Принтхит: очень крупные тиражи; забираем сами, всегда есть номер заказа, проблем с отгрузкой нет.
- Базовый офсетный партнёр для автосоздания заданий — **ГМ Принт** (в автоматике соответствует `partner_id = 1`).

### 2.2. Широкоформатная печать

- Текущий процесс: менеджер отправляет ТЗ партнёру, тот делает просчёт, затем PrintOffice добавляет свою наценку и отправляет цену клиенту.
- Цель — составить базу цен по широкоформатной печати, чтобы не запрашивать цену каждый раз, не тратить на это время и сразу делать наценку и называть клиенту сроки.
- Широкоформатный подрядчик по развёрнутому профилю — **Эдгар** (всегда забираем сами; `raw/crm-site-setup.md:181`). В автоматике широкоформат маршрутизируется на `partner_id = 2` — соответствие `partner_id = 2 ↔ партнёр` требует фиксации (см. ⚠️ в разделе 1). Ранее здесь ошибочно значились «FastPrint, PrintHit» — исправлено.

### 2.3. Сложная продукция (типография «Город»)

- Типография «Город» делает сложную продукцию: **печать браслетов, печать меню с лаком** (`raw/crm-site-setup.md:182`).
- Забираем всегда сами; всегда есть номер заказа, проблем с отгрузкой нет.

### 2.4. Сувенирная продукция и браслеты

- Печать браслетов и иной сувенирки PrintOffice также перезаказывает у подрядчиков (печать браслетов конкретно выполняет «Город», см. 2.3).
- Процесс: менеджер пишет подрядчику в Telegram, отправляет ТЗ → подрядчик называет стоимость и срок → PrintOffice добавляет наценку.

### 2.5. Правило формирования наценки (общее для подрядчиков)

К цене подрядчика добавляется наценка за вычетом всех расходов PrintOffice:

```
цена_клиенту = цена_подрядчика + наценка
наценка учитывает:
  - доставку от подрядчика в офис PrintOffice (или до клиента)
  - налоги 7 %
```

После согласования цены с клиентом берётся оплата, выполняется заказ, проверяется качество, продукция упаковывается и нумеруется по позициям.

### 2.6. Проблема запроса цен (болевая точка)

Запрос стоимости у подрядчика по каждому заказу отдельно (особенно для широкоформата и сувенирки) занимает время и тормозит расчёт. Требование: вести базу цен партнёров, чтобы рассчитывать стоимость и срок без ручного обращения к подрядчику.

## 3. Справочник партнёров и типы отгрузки

Курьер может ехать за заказом к партнёру (Эдгар, ГМ Принт, типография «Город»). У каждого партнёра свои адреса забора, инструкции и пропускной режим, поэтому нужен справочник партнёров.

Поля справочника партнёров:

- Название, контакты, адрес забора.
- Инструкция для курьера (как пройти, как проехать, нужен ли пропуск).
- Тип отгрузки: самовывоз, доставка партнёром, доставка нашим курьером.

В карточке заказа выбирается партнёр — после выбора автоматически подставляются инструкции для курьера.

## 4. Модуль «Партнёры» и личный кабинет партнёра

### 4.1. Назначение

Отдельная страница/портал, создаваемая в первую очередь для Эдгара (опционально для остальных партнёров). Доступ — по уникальной ссылке (токену) либо по логину/паролю.

### 4.2. Что видит партнёр

В ЛК отображаются только заказы, назначенные этому партнёру. Для каждого заказа показывается:

- Номер заказа (наш внутренний).
- Перечень позиций, материал, тираж.
- Адрес конечной доставки (куда отгрузить).
- Назначенный курьер (если вызываем мы) или статус «Заберут сами».

Возможности партнёра в ЛК:

- Отметить статус: «Принят в работу», «Готов к отгрузке», «Отгружен».
- Менять цену и срок по каждому заданию (изменения фиксируются в истории).

### 4.3. Уведомления партнёру

- При назначении нового заказа.
- При смене статуса доставки.

### 4.4. Интеграция с партнёрами без портала (ГМ Принт, Город, Виктория)

- CRM парсит входящие письма (от ГМ Принт) с номерами заказов и ссылками на отслеживание.
- Для остальных менеджер вручную меняет статус заказа («Передано партнёру», «Готов к отгрузке») на основе сообщений в мессенджере.

⚠️ ОТКРЫТЫЙ ВОПРОС: ГМ Принт фигурирует и как партнёр без портала (статусы ведутся через парсинг писем и вручную), и как партнёр с единым ЛК по токен-ссылке (см. макет ЛК с заголовком «ГМ Принт»). Нужно определить итоговый режим работы с ГМ Принт: портал/ЛК или парсинг почты + ручное ведение статусов.

## 5. Расширенная система работы с партнёром

### 5.1. Полный цикл (4 этапа)

1. **Автоматическое создание задания.** Клиент оформляет заказ на сайте (офсет/широкоформат) → система автоматически создаёт задание для партнёра → задание попадает в общий список партнёра.
2. **Партнёр видит все задания в одном ЛК.** Единая ссылка вида `https://partner.printoffice.ru/dashboard/{token}`; список всех заданий с фильтрацией по статусу; возможность менять цену в каждом задании.
3. **Отгрузка заказа.** Партнёр отмечает готовность → статус «ГОТОВ». Менеджер забирает заказ → статус «ОТГРУЖЕН». Информация об отгрузке попадает в ЛК партнёра.
4. **Интеграция с курьерскими службами.** Менеджер создаёт заявку на Достависта/Яндекс → система передаёт данные курьеру → ссылка на отслеживание попадает в ЛК партнёра.

### 5.2. Автоматическое создание задания

При оформлении заказа клиентом задание создаётся в зависимости от типа печати:

- `offset` → партнёр ГМ Принт (`partner_id = 1`).
- `wide` → партнёр по широкоформату (`partner_id = 2`).

```javascript
class AutoTaskCreator {
    async onOrderCreated(orderId) {
        const order = await this.getOrder(orderId);
        if (order.printing_type === 'offset') {
            await this.createPartnerTask(orderId, 1, order);   // ГМ Принт
        } else if (order.printing_type === 'wide') {
            await this.createPartnerTask(orderId, 2, order);   // широкоформат
        }
    }

    async createPartnerTask(orderId, partnerId, order) {
        const taskNumber = `TASK-${Date.now()}-${Math.floor(Math.random() * 1000)}`;
        const specifications = {
            product_type: order.product_type,
            format: order.format,
            material: order.material,
            color: order.color,
            quantity: order.quantity,
            width: order.width,
            height: order.height,
            area_sqm: order.area_sqm,
            services: order.services
        };
        await db.query(`
            INSERT INTO partner_tasks (
                order_id, partner_id, task_number, specifications, files,
                our_estimated_price, status, created_at
            ) VALUES ($1, $2, $3, $4, $5, $6, 'new', NOW())
        `, [orderId, partnerId, taskNumber, JSON.stringify(specifications),
            JSON.stringify(order.files), order.estimated_price]);
        await this.notifyPartnerNewTask(partnerId, taskNumber);
    }
}
```

## 6. Структура базы данных

### 6.1. Таблица `partners` (партнёры)

```sql
CREATE TABLE partners (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(255) NOT NULL,
    slug VARCHAR(100) UNIQUE,
    -- Контакты
    email VARCHAR(255),
    telegram VARCHAR(100),
    phone VARCHAR(50),
    -- Адрес самовывоза
    pickup_address TEXT,
    pickup_contact_person VARCHAR(255),
    pickup_phone VARCHAR(50),
    -- Реквизиты для оплаты
    payment_details JSON,
    -- Настройки
    notify_email BOOLEAN DEFAULT true,
    notify_telegram BOOLEAN DEFAULT true,
    -- Токен доступа к ЛК (единый для всех заданий)
    access_token VARCHAR(100) UNIQUE,
    token_expires_at TIMESTAMP,
    is_active BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
);
```

### 6.2. Таблица `partner_tasks` (задания для партнёров)

```sql
CREATE TABLE partner_tasks (
    id INT PRIMARY KEY AUTO_INCREMENT,
    order_id INT NOT NULL,
    partner_id INT NOT NULL,
    -- Номер задания
    task_number VARCHAR(50) UNIQUE NOT NULL,
    -- Параметры заказа
    product_type VARCHAR(100),
    specifications JSON NOT NULL,
    files JSON,
    -- Цены (партнёр может менять)
    our_estimated_price DECIMAL(10,2),
    partner_price DECIMAL(10,2),          -- партнёр может изменять
    final_price DECIMAL(10,2),
    -- Сроки
    partner_deadline DATE,
    -- Статусы
    status ENUM(
        'new',          -- новое (автоматически)
        'pending',      -- партнёр ответил
        'approved',     -- утверждено
        'paid',         -- оплачено
        'in_progress',  -- в работе
        'ready',        -- готово к отгрузке
        'shipped',      -- отгружено (забрали)
        'completed',    -- завершено
        'cancelled'     -- отменено
    ) DEFAULT 'new',
    -- Информация об отгрузке
    shipment_info JSON,                   -- {method, courier, tracking_link, shipped_at}
    -- Токен для доступа к конкретному заданию (устаревает, оставлен для совместимости)
    task_token VARCHAR(100),
    -- Комментарии
    manager_comment TEXT,
    partner_comment TEXT,
    -- Временные метки
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    responded_at TIMESTAMP,
    approved_at TIMESTAMP,
    paid_at TIMESTAMP,
    started_at TIMESTAMP,
    ready_at TIMESTAMP,
    shipped_at TIMESTAMP,
    completed_at TIMESTAMP,
    FOREIGN KEY (order_id) REFERENCES orders(id),
    FOREIGN KEY (partner_id) REFERENCES partners(id),
    INDEX idx_status (status),
    INDEX idx_partner (partner_id)
);
```

### 6.3. Таблица `partner_price_history` (история изменения цены партнёром)

```sql
CREATE TABLE partner_price_history (
    id INT PRIMARY KEY AUTO_INCREMENT,
    task_id INT NOT NULL,
    old_price DECIMAL(10,2),
    new_price DECIMAL(10,2),
    changed_by VARCHAR(50),                -- 'partner' или 'manager'
    changed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (task_id) REFERENCES partner_tasks(id)
);
```

### 6.4. Таблица `courier_orders` (курьерские заявки)

Курьерские заявки (связь с нашим заказом и заданием партнёра, служба `dostavista`/`yandex`, адреса забора/доставки, вес и габариты, статус `created`/`accepted`/`in_progress`/`completed`/`cancelled`, данные от службы и стоимость) хранятся в таблице `courier_orders`. Полное DDL — в модели данных: см. [data-model.md, раздел «courier_orders (курьерские заявки)»](../10-tech/data-model.md#courier_orders-курьерские-заявки).

## 7. Личный кабинет партнёра (единая ссылка)

ЛК доступен по единой токен-ссылке: `https://partner.printoffice.ru/dashboard/{token}`.

Блоки интерфейса ЛК:

- **Статистика.** Всего заданий / В работе / Готово / Завершено (пример: «Всего заданий: 45 | В работе: 3 | Готово: 2 | Завершено: 40»).
- **Фильтры.** По статусу, по периоду, поиск по тексту.
- **Список заданий.** Карточки заданий с разным состоянием.

Состояния карточек заданий:

| Состояние | Что показывается | Действия партнёра |
| --- | --- | --- |
| Новое (`new`) | Параметры заказа, наша ориентировочная цена; поля «Моя цена», «Срок (дней)», комментарий | СОХРАНИТЬ / ОТВЕТИТЬ |
| Готов (`ready`) | Параметры (продукция, материал, формат, тираж), цена и срок | ОТМЕТИТЬ ОТГРУЖЕННЫМ (после того как менеджер забрал) |
| Отгружено (`shipped`) | Дата и время отгрузки, курьер и телефон, ссылка на отслеживание | — (информационно) |

Пример карточки нового задания:

```
🆕 TASK-12345 | Получено: 14.04.2026
Офсетная печать | A5 | 130 г/м² | 4+4 | тираж 5000 шт
Наша ориентировочная цена: 15 000 руб
Моя цена: [ 12500 ] руб   📅 Срок: [ 3 ] дня   💬 [ Цена со скидкой ]
[ СОХРАНИТЬ ]  [ ОТВЕТИТЬ ]
```

Пример карточки готового задания (статус ГОТОВ):

```
✅ TASK-12344 | Статус: ГОТОВ
Широкоформатная печать | ПВХ 3 мм | A3 | 5 шт
Цена: 12 500 руб | Срок: 3 дня
[ ОТМЕТИТЬ ОТГРУЖЕННЫМ ]  — после того как менеджер забрал
```

Пример карточки отгруженного задания:

```
🚚 TASK-12343 | Статус: ОТГРУЖЕНО
Банер 2×1 м | Люверсы | Проклейка краёв
Отгружено: 14.04.2026 15:30
Курьер: Иван (Достависта), тел: +7 999 123 45 67
Отслеживание: https://dostavista.ru/order/12345
```

## 8. Класс управления партнёрским ЛК

Класс `PartnerDashboardManager` реализует создание доступа, выборку заданий, смену цены/срока с историей и отметку готовности.

```javascript
class PartnerDashboardManager {
    // Создание единого доступа для партнёра (токен на 1 год)
    async createPartnerAccess(partnerId) {
        const token = crypto.randomBytes(32).toString('hex');
        const expiresAt = new Date();
        expiresAt.setFullYear(expiresAt.getFullYear() + 1); // на год
        await db.query(`
            UPDATE partners SET access_token = $1, token_expires_at = $2 WHERE id = $3
        `, [token, expiresAt, partnerId]);
        const dashboardLink = `https://partner.printoffice.ru/dashboard/${token}`;
        return { token, dashboardLink };
    }

    // Получение всех заданий партнёра (с проверкой действительности токена)
    async getPartnerTasks(token, filters = {}) {
        const partner = await db.query(`
            SELECT id, name FROM partners
            WHERE access_token = $1 AND token_expires_at > NOW()
        `, [token]);
        if (partner.rows.length === 0) {
            throw new Error('Доступ недействителен или истек');
        }
        let query = `
            SELECT pt.*, o.order_number, o.client_name
            FROM partner_tasks pt
            JOIN orders o ON o.id = pt.order_id
            WHERE pt.partner_id = $1
        `;
        const params = [partner.rows[0].id];
        if (filters.status) {
            query += ` AND pt.status = $2`;
            params.push(filters.status);
        }
        query += ` ORDER BY pt.created_at DESC`;
        const tasks = await db.query(query, params);
        return { partner: partner.rows[0], tasks: tasks.rows, stats: this.calculateStats(tasks.rows) };
    }

    // Обновление цены/срока партнёром + запись в историю
    async updatePartnerPrice(taskId, token, newPrice, deadline, comment) {
        const task = await db.query(`
            SELECT pt.*, p.access_token
            FROM partner_tasks pt
            JOIN partners p ON p.id = pt.partner_id
            WHERE pt.id = $1 AND p.access_token = $2
        `, [taskId, token]);
        if (task.rows.length === 0) {
            throw new Error('Доступ запрещен');
        }
        const oldPrice = task.rows[0].partner_price;
        await db.query(`
            UPDATE partner_tasks
            SET partner_price = $1, partner_deadline = $2, partner_comment = $3,
                status = 'pending', responded_at = NOW()
            WHERE id = $4
        `, [newPrice, deadline, comment, taskId]);
        await db.query(`
            INSERT INTO partner_price_history (task_id, old_price, new_price, changed_by)
            VALUES ($1, $2, $3, 'partner')
        `, [taskId, oldPrice, newPrice]);
        await this.notifyManager(taskId, `Партнер изменил цену: ${oldPrice} → ${newPrice}`);
        return { success: true };
    }

    // Отметка готовности партнёром
    async markReady(taskId, token) {
        const result = await db.query(`
            UPDATE partner_tasks pt
            SET status = 'ready', ready_at = NOW()
            FROM partners p
            WHERE pt.id = $1 AND p.access_token = $2 AND pt.partner_id = p.id
            RETURNING pt.id
        `, [taskId, token]);
        if (result.rows.length === 0) {
            throw new Error('Доступ запрещен');
        }
        await this.notifyManager(taskId, 'Партнер отметил готовность заказа');
        return { success: true };
    }
}
```

## 11. Прозрачность загрузки

Прозрачность загрузки партнёра обеспечивается за счёт:

- единого ЛК со статистикой (всего заданий, в работе, готово, завершено);
- статусной модели задания (`new → pending → approved → paid → in_progress → ready → shipped → completed`, плюс `cancelled`);
- видимости адреса конечной доставки и назначенного курьера по каждому заданию;
- истории изменения цены/срока (`partner_price_history`), что делает прозрачными корректировки со стороны партнёра.

Кроме того, при планировании производства менеджер сверяет наличие материалов и загруженность производства — учёт загрузки является входным требованием к системе.
